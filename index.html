执行上下文

JS在第一次执行时会根据web引擎创建 全局上下文 在该上下文中运行JS脚本，在 调用函数时 从内存堆中提取函数 创建该函数的函数执行上下文 再将程序的执行转入 该函数上下文中 函数执行后再 返回全局上下文（这与C/C++的函数栈帧不同，反倒与thread机制类似，即进行上下文的保存和转换， 据此可推知： JS所实现的单线程异步是以此为基础实现的）

  • 关于JS中函数的存储

    是的，JS将定义的函数及其形参都按一定规则存入内存堆，在调用时从堆中提取。

调用栈

调用栈是JS引擎 跟踪 被调用函数所处位置的方式，JS使用其管理 执行上下文。

JS为每个被调用的函数创建一个 函数上下文 并依次压入调用栈中，由于JS为单线程语言，所以 只维护一个调用栈 当栈空后 脚本停止 。当执行的是一个递归函数，且陷入无限递归时会产生 栈溢出。

由于JS为单线程语言，只维护一个调用栈，所以 一次只执行一个操作 所有操作都是 同步的 而JS的 异步 特性可以在等待另一个任务完成时，执行其他任务。

Promise 对象

Promise 是一个对象，封装了 异步操作 的结果

Promise对象有一个状态，是以下三个之一

  • 等待中 | pending
  • 已完成，带有 值 | fulfilled
  • 由于 原因 而被拒绝 | rejected

初始时为 等待 ，之后根据运行结果，将状态更改为， 已完成 或 拒绝

使用 构造函数 Promise() 来创建 Promise对象

异步操作 成功执行 调用resolve回调函数，将对象状态改为已完成，并返回值

异步操作 被拒绝 调用reject回调函数，将状态对象改为被拒绝

一旦Promise 已完成 或 被拒绝 ，就 无法再 进入其他状态，实际上是被解析，只能从 等待 状态变为 已完成 或 被拒绝

Promise作为一个模板类，需要传入一个类型，该类型为数据的返回类型，即resolve中的value的类型

  • then()

  • catch()

    无论已完成或是被拒绝都执行相同代码

    接受一个包含可迭代对象的Promise

    即，所有传入的Promise对象被解析为fulfilled状态，该返回的数组中包含所有的解析后产生的结果。

    传入的iterable中某个Promise被拒绝，则返回一个被拒绝的Promise

    Promise.all() 起到一个 聚拢异步操作 的作用。

标志 RFlags寄存器

标志是独立于其他任何位的位(bit)信息，根据CPU的需要将相应位设置为0/1，以表示CPU内部某些条件的状态。
可以设置标志位作为向CPU发送信号的一种方式。

RFlags整体上是嵌在CPU内部的一个64位寄存器。是EFlags 32b位与Flags 16位的扩展。

RFlags目前仅使用了18位作为标志位，其余保留。

这18个标志位通常被分为三类：状态标志（Status Flags）、控制标志（Control Flags） 和 系统标志（System Flags）。

System V ABI 函数调用约定

  1. 整数/指针传递规则
    函数调用时，参数从左到右依次放入以下寄存器：

    参数位置 寄存器
    第1个 RDI
    第2个 RSI
    第3个 RDX
    第4个 RCX
    第5个 R8
    第6个 R9
    第7个及以上 通过栈传递（从右向左压栈）

  2. 返回值规则

    整数/指针返回值：存放在 RAX 中（64位）
    128位返回值（如long double或某些结构体）：使用 RAX 和 RDX 联合返回
    浮点返回值：使用 XMM0 寄存器（单/双精度浮点数）

  3. 保存责任规则（关键！）

    保存责任规则.png

    • 调用者保存寄存器：函数可以随意修改这些寄存器。如果调用者希望调用后这些寄存器的值保持不变，调用者需要在调用前保存它们。
    • 被调用者保存寄存器：如果被调用函数要使用这些寄存器，必须在函数开头保存它们的原始值，并在返回前恢复。

  4. 栈对齐要求

    在call指令执行前，栈指针RSP必须是16字节对齐的。因为call指令会将8字节的返回地址压栈，此时RSP会减去8，从而保证进入函数时RSP是16字节对齐的。

  5. 浮点/向量参数传递

    对于浮点数和向量参数，使用XMM0-XMM7寄存器（最多8个单/双精度浮点数参数）：

    • 第1个浮点参数：XMM0
    • 第2个浮点参数：XMM1
    • 以此类推…

  2. 为局部变量提供存储空间

    寄存器数量极其有限（仅16个通用寄存器）。任何复杂的函数（例如处理数组、大型结构体或递归函数）都需要在内存中存放局部变量。

  3. 传参数量超过6

    rdi rsi rdx rcx r8 r9

    第七个及其之后的参数需要从右到左压入栈中

    调用者会将它们压入自己的栈帧，而被调用函数则会通过自己栈帧指针 RBP 的一个固定偏移（如 [rbp+16]）来访问它们。这同样需要栈帧作为可靠的中转存储。

  4. 保证栈对齐

    出于性能和安全考虑，System V ABI 要求在调用函数之前，栈指针 RSP 必须16字节对齐。call 指令本身会压入一个8字节的返回地址，这会破坏对齐。因此，函数开场经常通过调整 RSP（例如 sub rsp, 8）来重新对齐，以便安全地调用其他函数（如 printf）。这个调整也是栈帧管理的一部分。

关于栈帧，类似的，像进程、线程等有进行上下文状态保存需要的结构，其实现基础仍是类似与函数栈帧的结构，是在此基础之上的。
函数栈帧是原子操作，线程是管理这些原子操作的执行流，进程是为这些执行流提供隔离资源的容器。 从保存几个寄存器到保存整个CPU上下文，思想一脉相承。这正是计算机系统中“局部性”和“分层”思想的优美体现。

转移指令及其原理

在 x86_64 汇编中，转移指令是控制程序执行流程的核心。它们可以分为四大类：

  1. 无条件转移指令
  2. 过程调用与返回指令
  3. 条件转移指令
  4. 特殊转移指令

一、无条件转移指令

这是最直接、最常用的跳转。

  • JMP (Jump)：无条件跳转到指定目标。
    • 作用：实现 goto、无限循环、跳出深层结构。
    • 示例：
      assembly jmp label ; 直接跳转到标签处 jmp [rax] ; 间接跳转，跳往地址存在 rax 指向的内存中 jmp rax ; 间接跳转，跳往地址存在 rax 寄存器中

二、过程（函数）调用与返回指令

这是实现函数模块化的基石。

  • CALL (Call Procedure)：
    • 作用：1) 将 RIP（下一条指令地址）压栈； 2) 跳转到目标函数。
    • 示例：call my_function
  • RET (Return from Procedure)：
    • 作用：从栈顶弹出地址到 RIP，从而返回到 call 之后的指令。
    • 示例：ret

三、条件转移指令（最常用、最复杂）

这是实现 if、else、while、for 等高级语言控制逻辑的关键。它们根据标志寄存器 (RFLAGS) 中的状态位来决定是否跳转。

最重要的四个状态位：

  • CF (Carry Flag)：无符号运算溢出/借位。
  • ZF (Zero Flag)：结果为零。
  • SF (Sign Flag)：结果为负（有符号数）。
  • OF (Overflow Flag)：有符号运算溢出。

下表列出了最常用的条件跳转指令及其判断逻辑：

指令 含义 检查的标志位（条件） 典型应用场景
JE / JZ 等于 / 为零时跳转 ZF = 1 if (a == b)，循环结束判断
JNE / JNZ 不等于 / 不为零时跳转 ZF = 0 if (a != b)，循环继续判断
JG / JNLE 大于跳转（有符号） ZF = 0 and SF = OF if (a > b) （有符号数比较）

在源文件中使用动态链接过程时 需要以下方法进行相对位置调用

wrt 与 plt

  • wrt 是 NASM 的关键字，全称 “With Respect To”，表示“相对于……”。它用来指定一个符号的重定位方式。
  • ..plt 是一个特殊的段（section）或符号，代表过程链接表（Procedure Linkage Table）。PLT 是动态链接机制的一部分，用于延迟绑定外部函数（如 C 库函数 printf、print 等）。

为什么需要 wrt ..plt？

在动态链接的可执行文件或共享库中，对外部函数的调用地址在链接时无法确定（因为函数在另一个共享库中）。直接写 call print 会生成一个对 print 符号的绝对地址引用，这会导致链接器在生成共享库时产生需要重定位的代码段，从而触发你之前遇到的警告（relocation in read-only section）。

使用 call print wrt ..plt 后，汇编器生成的指令会跳转到 PLT 中与该函数对应的桩代码（stub）。PLT 的代码在运行时由动态链接器负责填充正确的函数地址，从而：

  1. 保持代码段只读：PLT 本身在代码段，但其内部的重定位条目指向 GOT（全局偏移表），GOT 是可写的。
  2. 实现延迟绑定：第一次调用时解析函数地址，后续调用直接使用已解析的地址。
  3. 满足位置无关代码（PIC）要求：避免了在只读代码段中插入绝对地址重定位。

何时必须使用 wrt ..plt？

  • 在共享库（.so）的代码中，对外部函数（定义在其他库中）的调用必须通过 PLT，否则会产生不可重定位的代码，导致共享库无法正常加载或触发警告。
  • 在可执行文件中，如果可执行文件是动态链接的且调用了外部库函数，链接器通常会自动处理 PLT 重定位，因此显式写 wrt ..plt 并非总是强制，但显式指定能让代码意图更清晰，并避免某些 NASM 模式下的默认行为问题。

补充：..got 和 ..plt 的关系

  • ..plt 用于函数调用。
  • ..got 用于访问外部全局变量。例如 mov rax, [rel myvar wrt ..got] 会从 GOT 中获取变量 myvar 的实际地址。

二、重复前缀

重复前缀使字符串指令可以自动循环执行，从而处理整个内存块。常用前缀有：

前缀 含义 适用指令
REP repeat 重复执行直到 RCX 变为 0 MOVS、STOS、LODS
REPE / REPZ repeat equal/zero 相等时重复（ZF=1 且 RCX≠0） CMPS、SCAS（用于查找不相等）
REPNE / REPNZ repeat not equal/zero 不相等时重复（ZF=0 且 RCX≠0） CMPS、SCAS（用于查找相等）

工作原理：每次执行指令后，RCX 减 1，并根据指令类型和前缀条件判断是否继续。同时，RSI 和 RDI 会根据方向标志自动增减。

三、方向标志 DF

方向标志 DF 控制指针的增减方向：

  • DF=0（使用 CLD 清除）：指针递增（向高地址移动）。clean direction flag
  • DF=1（使用 STD 置位）：指针递减（向低地址移动）。set direction flag

在操作字符串时，务必根据需要设置正确的方向。

七、字符串指令详解

字符串指令是 x86/x86-64 汇编中用于处理连续内存块（字符串或数组）的专用指令，它们与重复前缀配合可以高效地完成批量内存操作。下面详细描述每个核心字符串指令与不同重复前缀组合时的功能，并说明其执行过程。

一、字符串指令基础

所有字符串指令都会根据方向标志 DF 自动更新源指针 RSI 和目标指针 RDI（或仅其中一个）：

  • DF=0（CLD）：指针递增（向高地址移动）。
  • DF=1（STD）：指针递减（向低地址移动）。

更新的步长由指令操作数大小决定：

  • B（字节）：±1
  • W（字）：±2
  • D（双字）：±4
  • Q（四字）：±8

重复前缀 REP、REPE/REPZ、REPNE/REPNZ 使指令在 RCX ≠ 0 的条件下循环执行，每次执行后 RCX 自动减 1，并根据前缀条件决定是否继续。

二、各指令与重复前缀的组合功能

1. MOVS（Move String）

功能：从 [RSI] 复制一个元素到 [RDI]，并更新两个指针。

组合 功能描述 类比 C 语言操作
REP MOVS 将 RCX 个元素从源地址复制到目标地址。 memcpy(dest, src, count * element_size)
无其他前缀 （无条件重复） -

执行细节：

  • 每复制一个元素，RSI 和 RDI 根据 DF 增减，RCX 减 1。

  • 当 RCX 变为 0 时停止。

2. STOS（Store String）

功能：将累加器（AL/AX/EAX/RAX）的值存储到 [RDI]，并更新目标指针。

组合 功能描述 类比 C 语言操作
REP STOS 用累加器的值填充 RCX 个目标元素。 memset(dest, value, count * element_size)
无其他前缀 （无条件重复） -

执行细节：

  • 每次存储后 RDI 更新，RCX 减 1，直到 RCX=0。

  • 累加器值保持不变。

3. LODS（Load String）

功能：从 [RSI] 加载一个元素到累加器，并更新源指针。

组合 功能描述 类比 C 语言操作
REP LODS 连续加载 RCX 个元素到累加器，但最终只保留最后一个值。 无直接对应，实际很少用
无其他前缀 （无条件重复） -

执行细节：

  • 每次加载后累加器被覆盖，RSI 更新，RCX 减 1。

  • 循环结束后，累加器存放最后一个元素的值，之前的全部丢失。

  • 因此 REP LODS 实用价值很低，通常配合其他指令在循环中使用。

4. CMPS（Compare String）

功能：比较 [RSI] 和 [RDI] 的两个元素，影响标志位，并更新两个指针。

组合 功能描述 类比 C 语言操作
REPE CMPS / REPZ CMPS 当两个元素相等时继续比较，直到遇到不相等的元素或 RCX=0。 memcmp 的前缀检测，或查找第一个不同处
REPNE CMPS / REPNZ CMPS 当两个元素不相等时继续比较，直到遇到相等的元素或 RCX=0。 查找第一个相同处

执行细节：

  • 每次比较后更新指针和 RCX，并根据比较结果设置 ZF。

  • REPE：循环继续的条件是 ZF=1（相等）且 RCX≠0。

  • 若因 ZF=0（不相等）停止，则 RSI/RDI 指向不相等的位置。

  • 若因 RCX=0 停止且一直相等，则 ZF=1 表示全部相等。

  • REPNE：循环继续的条件是 ZF=0（不相等）且 RCX≠0。

  • 若因 ZF=1（相等）停止，则指针指向相等的位置。

  • 若因 RCX=0 停止且从未相等，则 ZF=0。

5. SCAS（Scan String）

功能：将累加器（AL/AX/EAX/RAX）与 [RDI] 的元素比较，影响标志位，并更新目标指针。

组合 功能描述 类比 C 语言操作
REPE SCAS / REPZ SCAS 当累加器与目标元素相等时继续扫描，直到遇到不相等或 RCX=0。 查找第一个不等于给定值的元素
REPNE SCAS / REPNZ SCAS 当累加器与目标元素不相等时继续扫描，直到遇到相等或 RCX=0。 查找第一个等于给定值的元素（如 strchr）

执行细节：

  • 每次比较后更新 RDI 和 RCX，并根据比较结果设置 ZF。

  • REPE：继续条件为 ZF=1（相等）且 RCX≠0。

  • 若因 ZF=0 停止，RDI 指向第一个不相等的位置，且 ZF=0。

  • 若因 RCX=0 停止，则所有元素都与累加器相等，ZF=1。

  • REPNE：继续条件为 ZF=0（不相等）且 RCX≠0。

  • 若因 ZF=1 停止，RDI 指向第一个相等的位置，且 ZF=1。

  • 若因 RCX=0 停止，则未找到相等元素，ZF=0。

  • 确保 RCX 存放的是元素个数，而非字节数（除非使用字节指令）。

若在设计 多线程类 时采用 内部锁定 ，则会使自己面临 效率低下 或 死锁 的风险。若依赖于调用者提供的锁定，则会使类难以使用。而使用外部锁定的方法，将锁留给客户端代码就可以避免以上问题。

互斥锁概念

互斥锁对象有助于 防止数据竞争 ，并允许线程之间的数据进行 线程安全同步 。线程通过调用锁函数之一 获得 互斥对象的 所有权 ，并通过调用相应的解锁函数 放弃 所有权 。
Boost.Thread 提供具有独占所有权语义的 递归 和 非递归 互斥锁，以及 共享所有权（多读、单写） 互斥锁。
以下的概念中都有集成有 CPP类型萃取 进行类型分析，但在笔记中并不详述。命名都是 **is_**格式

  • 互斥量的关键设计点

    它协调的是不同线程对同一资源的访问，而不是跟踪哪个具体线程应该访问。所有权机制确保同一时间只有一个线程可以进入临界区，无论这个线程是谁。

  • Boost.Thread 支持锁对象的四个基本概念：

    • Lockable
    • TimedLockable
    • SharedLockable
    • UpgradeLockable

    每种互斥锁类型都实现以上概念中的一个或多个，各种锁类型也是如此。

  • BasicLockable 概念

    <boost/thread/lockable_concepts.hpp>

    此概念，对 独占所有权进行建模 。如果以下表达式格式良好并具有指定语义，则该类型满足 BasicLockable 要求。

    通过 lock() 或的所有权必须通过 unlock() 释放

    • obj.lock();

      通过该调用 获得 锁的所有权

      要求 如果互斥锁不是递归的，则调用线程不拥有互斥锁。（调用线程：当前正在执行该函数的线程）即，不能在同一线程中连续重复获取互斥锁。

      效果 当前线程阻塞，直到可以为当前线程获得所有权。即，阻塞结束后才可能获取互斥锁。

      同步 对同一对象执行先前的 unlock() 操作与此操作同步。即，线程获取的锁的所有权在使用后需要主动释放。

    • obj.unlock();

      通过该调用 释放 锁的所有权

      要求 当前线程拥有 obj 。

      同步 此操作与后续获得同一对象所有权的锁操作同步。

      效果 释放 当前线程对 obj 的 锁定

  • Lockable 概念

    若类型满足 Lockable 的要求，则其也 必须满足 BasicLockable 的要求，并且以下表达式形势良好且具有指定的语义。

    • obj.try_lock()

      通过该调用获得锁的所有权，且必须通过调用 unlock() 释放

      要求 若互斥锁不是递归的，则调用线程不拥有互斥锁。

      效果 尝试 为当前线程获取所有权而 不阻塞 。即，仅仅是立即判断是否可以获取锁，若可，则获得锁；若非，则不获取锁，同时不阻塞，继续执行可用下文。

      同步 若 try_lock() 返回 true ，则对同一对象先前的 unlock() 操作与此操作同步。

      注意 由于 unlock() 不与随后失败的 try_lock() 同步，因此可见性规则非常弱，以至于在获取失败后几乎无法了解状态，即使在没有虚假失败的情况下也是如此。

  • TimedLockable 概念

    TimedLockable 概念细化了 Lockable 概念，以添加在尝试获取锁时对超时的支持。

    变量

    • obj 为互斥量类型的值
    • rel_time 表示 chrono::duration 的实例
    • abs_time 表示 chrono::time_point 的实例

    表达式

    • obj.try_lock_for(rel_time)

      要求 锁互斥锁不是递归的，则调用线程不拥有互斥锁。

      效果 尝试为当前线程获取所有权。 阻塞 ，直到可以 获取所有权 或 超过了指定等待的时间 。

      同步 若返回 true 则对同一对象先前的 unlock() 操作同步。

    • obj.try_lock_until(abs_time)

      要求 若互斥量不是递归的，则调用线程不拥有互斥锁。

      效果 尝试为当前线程获取所有权。 阻塞 ，直到可以 获取所有权 或 达到指定时间 。

      同步 若返回值为 true 则对同一对象先前的 unlock() 操作同步。

    • obj.timed_lock(rel_time)

    • obj.timed_lock(abs_time)

  • SharedLockable 概念

    之前的 lock() unlock() 等方法都是 独占锁 写者锁
    以下的 *_shared() 方法都是 共享锁 读者锁

    SharedLockable 概念 是 TimedLockable 概念 的改进， 允许共享所有权 和 独占所有权 。是标准的 多读/单写 模型：最多只能有一个线程拥有独占所有权，如果任何线程拥有 独占所有权 ，则其他线程 不能拥有共享或独占所有权 。或者，许多线程可以拥有共享所有权。

    如果类型满足 SharedLockable 的要求，则其必须满足 TimedLockable 的要求，并且以下表达式格式良好且具有指定的语义。

    变量

    • obj 表示类型的值
    • rel_time 表示 chrono::duration 的实例
    • abs_time 表示 chrono::time_point 的实例

    表达式

    • obj.lock_shared()

      效果 当前 线程阻塞 ，直到可以为当前线程获得共享所有权。

      后置条件 当前线程拥有 obj 的共享所有权。

    • obj.try_lock_shared()

      效果 尝试在 不阻塞 的情况下为当前线程获取共享所有权。

      后置条件 如果调用返回 true ，则当前线程拥有 obj 的共享所有权。

    • obj.try_lock_shared_for(rel_time)

      效果 尝试为当前线程获取共享所有权。 阻塞 直到获得共享所有权，或者经过指定的时间。如果指定的时间已经过去，则表现为 try_lock_shared() 。

    • obj.try_lock_shared_until(abs_time)

      效果 尝试为当前线程获取共享所有权。 阻塞 直到获得共享所有权，或者达到指定的时刻。若指定的时刻已经过去，则表现为 try_lock_shared() 。

    • obj.unlock_shared()

      先决条件 当前线程拥有 obj 的共享所有权。

      效果 释放当前线程对 obj 的共享所有权。

      后置条件 当前线程不再拥有 obj 的共享所有权。

    • obj.timed_lock_shared(abs_time)

      效果 尝试为当前线程获取共享所有权。 阻塞 直到获得共享所有权，或达到指定的时刻。如果指定的时刻已经过去，则表现为 try_lock_shared() 。

  • UpgradeLockable 概念

    UpgradeLockable 概念 是对 SharedLockable 概念的改进，允许 可升级所有权 以及 共享所有权 和 独占所有权 。

    单个线程可以拥有 可升级所有权 ，同时 其他线程可以拥有 共享所有权 。

    可升级所有权线程，可以随时尝试将其所有权升级为 独占所有权 。若没有其他线程拥有共享所有权，则升级立即完成，此时必须通过调用 unlock() 来释放锁。

    若在其他线程拥有共享所有权是进行升级，则尝试将失败，并且该线程将被 阻塞 ，直到可以获取 独占所有权 。

    所有权可以被 降级 或 升级 。

    共享所有权 → 可升级所有权 → 独占所有权

    类型满足 UpgradeLockable 要求，则它满足 SharedLockable 要求，并且以下表达式格式良好且具有指定的语义。

    通过调用 lock_upgrade() 获取的锁所有权必须通过调用 unlock_upgrade() 来释放。如果通过调用 unlock_xxx_and_lock_yyy() 函数更改了所有权类型，则必须通过相应的所有权释放函数进行所有权释放。

    • obj.lock_upgrade()

      先决条件 调用线程不拥有互斥锁

      效果 当前线程 阻塞 ，直到可以为当前线程获得升级所有权。

    • obj.unlock_upgrade()

      先决条件 当前线程拥有 obj 的升级所有权。

      效果 释放当前线程对 obj 的升级所有权

      同步 此操作与后续获得同一对象所有权的锁操作同步。

条件变量

<boost/thread/condition_variable.hpp>

类 condition_variable 和 condition_variable_any 提供了一种机制，允许一个线程 等待 另一个线程通知某个特定条件已变为真。

一般用法是，一个线程锁定一个互斥量，然后在 condition_variable 或 condition_variable_any 的实例上调用 wait 。当线程从 等待中唤醒 时，它会检查相应的条件是否现在为真，若是，则继续。若非，则调用 wait() 继续等待。有两种返回值，bool cv_stat枚举量。

注意，lock被传递给wait，wait原子的将线程添加到等待条件变量的线程集合中，并解锁互斥锁。当线程被唤醒时，互斥锁将在调用wait返回之前再次被锁定。这允许其他线程获取互斥锁以更新共享数据，并确保与条件相关联的数据被正确同步。

同时，另一个线程将条件设置为 true ，然后在条件变量上调用 notify_one 或 notify_all 以分别唤醒（通知）一个等待线程或所有等待线程。

同步原语 wait 与 线程操作 yeild 的区别在于：
yeild 是由线程自身让出时间片给其他线程。是一种 就绪状态 。仍消耗CPU周期。
wait 是为了等待特定的同步而主动进入阻塞状态。是一种 等待/阻塞状态 。CPU周期使用为零。

条件变量的同步原语 wait 中可以传入一个返回值为 bool 的谓词。
注意，在更新共享数据之前，会锁定一个互斥锁，但互斥锁不必再调用 notify_one 时保持锁定。

屏障（barrier）

<boost/thread/barrier.hpp>

屏障是一个简单的概念，也被称为 汇合点 ，它是 多个线程之间的同步点 。

屏障配置为 特定数量的 线程 ，并且当线程到达屏障时，它们必须等待直到所有的线程都到达。一旦最后一个线程到达屏障，所有等待线程都可以继续，并且屏障会被 重置 。

boost::barrier 的实例 不可复制或移动

    效果 构造一个用于 count 个线程的屏障

    要求 完成函数调用 completion() 的结果类型是 void 或 unsigned int 。

    效果 构造一个用于 count 个线程和一个完成函数 completion 的屏障。

    先决条件 没有线程在 *this 上等待。

    效果 销毁*this

    重置取决于屏障是否使用 完成函数构造 。如果没有完成函数或者完成函数的结果为 void ，则重置包括恢复原始计数。否则，重置包括分配完成函数的结果（必须不为0）。

    同步点 确保多个线程在特定点同步
    阶段控制 协调多阶段并行算法的执行
    数据一致性 在关键操作前确保所有线程就绪

Futures

futures 库提供了一种处理同步 future 值的方法，无论这些值是由另一个线程生成，还是在单个线程上相应外部刺激，或按需生成。

boost::promise boost::futures boost::packaged_task 是对上述同步原语的更高级别的抽象。用于简化并发编程，特别是异步操作和线程间传递结果。

通过以下四个类模板来实现：

  • future
  • boost::shared_future 用于检索异步结果
  • boost::promise
  • boost::packaged_task 用于生成异步结果

future 的一个实例拥有 对结果的唯一引用 。

所有权可以在实例之间使用 移动构造函数 或 移动赋值运算符 进行转移，但最多只有一个实例拥有对 给定异步结果的引用 。当结果准备好时，它会通过 右值引用 从 boost::future::get() 返回，以便可以根据类型适当地 移动 或 复制 结果。

另一方面， boost::shared_future 的多个实例 可能引用相同的结果 。实例可以自由复制和分配，并且 boost::shared_future::get() 返回 const 引用，以便可以安全地多次调用 boost::shared_future::get() 。可以将 future 的实例移动到 boost::shared_future 的实例中，从而转移关联的异步结果的 多有权 ，但反之则不可。

boost::async 是一种运行异步任务的简单方法。调用 boost::aync 会返回一个 future ，该future将包含任务的结果。

可以单独使用future，也可以使用 boost::wait_for_any() 和 boost::wait_for_all() 函数之一等待。

  • 创建异步值

    可以使用 boost::promise 或 boost::packaged_task 设置 future 中的值。

    boost::packaged_task 是一个可调用对象，它包装一个函数或可调用对象。调用 packaged_task 时，它会 依次调用包含的函数 ，并使用返回值填充 future 。

    boost::promise 稍微低级一些：它只提供显式函数来将值或异常存储在关联的future中。因此，当值可能 来自多个可能的源 ，或者 单个操作可能产生多个值 时，可以使用promise。

  • 等待回调和延迟 Future

    boost::promise 和 boost::packaged_task 都支持 等待回调 ，当线程在对 wait() 或 timed_wait() 的调用中 阻塞时 ，会调用该回调，该回调发生在等待来自 boost::promise 或 boost::packaged_task 的结果的future上时，该回调在执行等待的线程中。可以使用相应的 boost::promise 或 boost::package_task 上的 set_wait_callback() 成员函数来设置这些回调。

    这允许 延迟future ，其中直到某些线程需要结果时才实际计算结果。

    上述示例中， f.get() 的调用会回调 invoke_lazy_task ，该回调运行任务以设置该值。如果删除对 f.get() 的调用，则永远不会运行该任务。

  • 处理分离线程和线程特定变量

    分离线程给具有线程存储持续时间的对象带来了问题。如果使用除了 boost::thread::join() 的机制来等待 线程完成工作，如，等待future准备就绪，那么线程特定变量的析构函数 仍然会在等待线程恢复后运行 。方法是 确保在future 准备就绪之前 销毁具有线程存储持续时间的对象。

  • 异步执行

    boost::async 提供了一种简单的方式来运行异步任务，以利用可用的硬件并发性。

    调用 boost::async 会返回一个 boost::future ，该future将包含任务的结果。根据启动策略，任务可以在其自身的线程上异步运行，也可以在调用该future的wait() 或 get()成员函数的线程上同步运行。

    启动策略可以是 boost::launch::async（要求运行时创建一个异步线程） 或 boost::launch::deferred（表示只想函数调用延迟到稍后的时间，即惰性求值） 。此参数是可选的，若省略，函数将使用默认策略。

撤销修改

可以使用
git checkout -- file
以丢弃 工作区 的更改。即，将 file 在工作区的修改都丢弃。 会让仓库回到 最近一次commit或add时的状态 。

可以使用
git reset HEAD filename
将 暂存区 的修改撤销。

远程仓库

Git仓库和GitHub仓库之间的传输是通过 SSH 进行加密的，所以需要一些设置。

  1. 创建SSH Key 。在当前的用户主目录下，可以查看是否有 .ssh目录 ，若存在，则可以查看该目录下是否有 id_rsa 与 id_rsa.pub 文件，若存在则可以直接从第二步开始。
    若不存在该目录与文件，可以创建SSH Key:
    ssh-keygen -t rsa -C "email@example.com"
    将email@example.com换为自己的邮箱地址。
    id_rsa 是 私钥
    id_rsa.pub 是 公钥

  2. 在GitHub中添加SSH密钥 。在 Account/SSH and GPG keys 中，点击 New SSH key 加入SSH公钥（id_rsa.pub文件中的内容）。

添加远程库

若在本地创建了一个Git仓库后，又想在GitHub上创建一个Git仓库，并让两个仓库进行远程同步，可以创建远程仓库。

  1. 先在GitHub上通过 Create a new repo 创建一个Repository
  2. 再在本地仓库中运行以下命令：
    git remote add origin git@github.com:iLogIng/RepoName.git
  3. 添加后，远程库的名字就是 origin ，可以进行更名。
  4. 可以将本地库的所有内容推送到远程库中：
    git push -u origin master
    实际上是使用 push 将当前分支 master 推送到远程库。
    在第一次推送时，远程库为空，需要加上 -u 选项。
    接下来，只要本地做了提交，就可以通过命令：
    git push origin master
    将本地 master 分支的最新修改推送至GitHub 。

删除远程库
若添加的时候地址写错，或是想要删除远程库，可以使用
git remote rm <name>
命令。，使用前，可以先用
git remote -v
查看远程库的信息
然后根据名字删除，如删除 origin：
git remote rm origin
此处的删除实际上是接触了本地和远程的绑定关系，并不是物理上删除了远程库。若要真正的删除远程库上的内容，则需要在GitHub上进行网页操作。

从远程库克隆

可以使用命令
git clone git@github.com:iLogIng/Repo-name.git
在本地克隆一个远程库。
