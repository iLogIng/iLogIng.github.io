基础版本控制

该笔记操作设定在以下仓库中进行
mkdir test-dir

  1. 添加仓库
    使用 git init 进行仓库的初始化，此时文件夹 test-dir/ 已经绑定到一个 git Repo 。
    ./test-dir $: git init

  2. 将文件添加到版本库

    1. 使用 git add [filename] 将[filename]文件加入版本库。
      ./test-dir $: git add README.md
    2. 使用 git commit -m “describe” 将添加到版本库中的文件提交到仓库，其提交描述为 “describe” 。
      ./test-dir $: git commit -m "Add the README.md file into git Repo"
    • commit 相当于为仓库添加一个 版本快照 。
    • commit 对版本快照进行管理。

  3. 查看仓库状态
    使用 git status 查看仓库状态，此时输出仓库状态信息。
    ./test-dir $: git status

查看已修改但是未提交的内容
使用 git diff 查看未提交的修改内容。
./test-dir $: git diff

  1. 更换快照版本
    可以使用 git log 命令可以查看最近到最远的 提交日志 。
    可以使用
    git reset --hard commit_id
    该命令可以回到上一个版本的 已提交状态 。
    • –soft 回到上个版本的 未提交状态 。
    • –mixed 回退到上个版本的 已添加未提交状态 。
    • commit_id 可以是一个 快照id 也可以是 HEAD^ 表示上个快照版本 HEAD^^ 表示向上数的第二个快照版本 HEAD~N 表示向上数第N个快照版本。
    • commit_id 可以填写部分 commit_id 的内容，系统可以自动检查。
      可以使用 git reflog 命令可以查看所有的时间从近到远的 所有提交日志 可以找到过去的提交内容。

工作区和暂存区

工作区

就是一个文件夹，就是工作文件夹

暂存区

工作区有一个名为 .git 的隐藏目录，不是工作区，而是作为Git的版本库。

git add filenames 实际上是将 filenames 加入到暂存区。
git commit -m “describe” 实际上是将 暂存区 的内容添加到当前分支。

管理修改

git 管理的是 修改 而非 文件

可以使用
git diff HEAD -- filename
来对比当前工作区和版本库中的内容。

撤销修改

可以使用
git checkout -- file
以丢弃 工作区 的更改。即，将 file 在工作区的修改都丢弃。 会让仓库回到 最近一次commit或add时的状态 。

可以使用
git reset HEAD filename
将 暂存区 的修改撤销。

远程仓库

Git仓库和GitHub仓库之间的传输是通过 SSH 进行加密的，所以需要一些设置。

  1. 创建SSH Key 。在当前的用户主目录下，可以查看是否有 .ssh目录 ，若存在，则可以查看该目录下是否有 id_rsa 与 id_rsa.pub 文件，若存在则可以直接从第二步开始。
    若不存在该目录与文件，可以创建SSH Key:
    ssh-keygen -t rsa -C "email@example.com"
    将email@example.com换为自己的邮箱地址。
    id_rsa 是 私钥
    id_rsa.pub 是 公钥

  2. 在GitHub中添加SSH密钥 。在 Account/SSH and GPG keys 中，点击 New SSH key 加入SSH公钥（id_rsa.pub文件中的内容）。

添加远程库

若在本地创建了一个Git仓库后，又想在GitHub上创建一个Git仓库，并让两个仓库进行远程同步，可以创建远程仓库。

  1. 先在GitHub上通过 Create a new repo 创建一个Repository
  2. 再在本地仓库中运行以下命令：
    git remote add origin git@github.com:iLogIng/RepoName.git
  3. 添加后，远程库的名字就是 origin ，可以进行更名。
  4. 可以将本地库的所有内容推送到远程库中：
    git push -u origin master
    实际上是使用 push 将当前分支 master 推送到远程库。
    在第一次推送时，远程库为空，需要加上 -u 选项。
    接下来，只要本地做了提交，就可以通过命令：
    git push origin master
    将本地 master 分支的最新修改推送至GitHub 。

删除远程库
若添加的时候地址写错，或是想要删除远程库，可以使用
git remote rm <name>
命令。，使用前，可以先用
git remote -v
查看远程库的信息
然后根据名字删除，如删除 origin：
git remote rm origin
此处的删除实际上是接触了本地和远程的绑定关系，并不是物理上删除了远程库。若要真正的删除远程库上的内容，则需要在GitHub上进行网页操作。

从远程库克隆

可以使用命令
git clone git@github.com:iLogIng/Repo-name.git
在本地克隆一个远程库。

Makefile

在Linux环境下，常常使用make程序自动化变异大型源码。
项目提供一个Makefile文件，make以此为基础进行程序的编译和链接。
make并不限定为构建C/C++项目服务，还可以用于任何文件的组织与处理，但主要用于Unix/Linux环境的自动化开发。

GNU make

Makefile基础

在Linux环境下，输入make命令，程序在当前目录下查找一个名为Makefile的文件，然后，根据该文件定的规则，自动化的执行任意命令。

规则

Makefile文件由若干条规则(Rule)构成，每一条规则指出一个目标文件(Target)，若干依赖文件(prerequisites)，以及生成目标文件的命令。

Makefile定义了一系列规则，每个规则在满足依赖文件的前提下执行命令，就能创建一个目标文件。

伪目标

对于中间的生成文件，可以安全的进行删除。
可以使用clean规则进行中间生成产物的删除。

clean规则没有依赖文件，要执行clean，必须用命令make clean
在执行clean时，并没有创建一个名为clean的文件。若手动创建一个clean文件，该clean规则就不会执行。

一般并不需要用.PHONY标识clean等约定俗成的伪目标名称。

控制打印

默认情况下，make会打印出它执行的每一条命令。如果不想打印某一条命令，可以在命令前加上@，表示命令执行时，不打印该命令。

控制错误

make在执行命令时，会检查每一条命令的返回值，如果返回错误(非0值)，就会中断执行。

编译C程序

C/C++程序的编译过程

main.cpp 源文件
| 预处理器
main.i 被预处理修改了的源文件
| 编译器
main.s 汇编程序
| 汇编器
main.o 可重定向的目标文件
| 链接器
main 可执行文件

以上过程可简化为以下处理步骤：

  1. 将每个.c或.cpp文件编译为.o文件
  2. 将所有.o文件链接为最终的可执行文件

自动变量

在Makefile中，经常可以看到$@ $<这样的变量，这种变量被称为 自动变量(Automatic AVariable)

  • $@ 当前规则的目标名 在app: main.o规则中，$@就是app
  • $< 第一个依赖项名 在app: main.o util.o规则中，$<就是main.o
  • $^ 所有依赖项列表 在app: main.o util.o规则中，$^就是main.o util.o
