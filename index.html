事先声明

  1. AI 是 外置大脑 而非个人的替身

    • 要保证项目代码的最终决策权在我们自己，你需要不断引导它分析约束、评估风险、给出可选方案，而不是直接接受第一个答案。

  2. 好的架构是分视图、分阶段、分模块的

    • 你必须在提示词里显式要求 AI 从不同架构视图（逻辑、过程、开发、部署、数据）思考，避免它过早陷入细节。
    • 你最好能事先规范化项目的顶层设计，再与AI进行总体的设计讨论，逐步深入。

  3. 每一次讨论都是一次 架构评审

    • 把提示词设计成微小的评审会议：输入上下文、假设，要求 AI 输出分析、质疑、建议。
    • 但是，绝不能被AI牵着鼻子走，在整理AI的建议时需要自己的思考，纠正AI偏离预期设计的想法。

C++ Boost::Log

发表于 2026-04-03 更新于 2026-04-10

Log

一个高度模块化和可扩展的C++日志库，旨在为应用程序提供灵活的日志记录功能。

Boost::Log Refference

设计理念

提供 高度模块化 和 可扩展 的日志处理。各个模块可以独立使用和替换，并且能够添加自定义功能。支持宽/窄字符记录。

三层架构

  1. 日志数据收集层
    • 负责从程序中处理收集的日志事件
  2. 收集数据处理层
    • 负责处理收集到的日志事件
  3. 连接前两层的中央枢纽
    • 协调管理日志在上两层之间的流转

设计要点： 这个架构的核心优势是正交性，即日志的收集、处理方式和日志内容本身是相互独立的。这种设计让库不仅能用于传统日志记录，还能用于向用户发出通知、累积统计信息等场景。

关键组件

构成以上三层的具体组件：

  • 日志源 (Logging Sources)： 日志事件的发起者 。最常见的是日志记录器 (Logger)，它提供了类似 std::ostream 的流式接口来 格式化 日志消息。库也支持其他类型的源，如捕获子进程的控制台输出等。

  • 属性与属性值 (Attributes and Attribute Values)： 日志记录的元数据 。一个属性本质上是获取某个值的函数（例如获取当前时间），其返回的值就是 属性值 ，会在日志处理的后续阶段使用。属性分为 三种 类型：

    • 全局属性 ：附加到 所有 日志记录，优先级最低。

    • 线程特定属性 ：仅附加到 特定线程 生成的记录。

    • 源特定属性 ：由日志源维护 ，仅附加到通过 该源生成的记录 ，优先级最高。

    • 当同名属性出现在多个集合中时，优先级高的属性会覆盖优先级低的。

  • 日志核心 (Logging Core)： 中央枢纽的具体实现。它负责接收日志事件，并应用全局过滤器来决定哪些日志记录可以继续传递。核心被实现为 单例 (Singleton) ，可通过 boost::log::core::get() 访问。

  • 过滤器 (Filters)： 核心的一部分功能。过滤器用于在核心层面 筛选 日志记录，决定其是否被处理。

  • 接收器 (Sinks)： 日志处理的最终目的地。它是处理层的核心，负责将日志记录 格式化并输出到文件 、控制台、网络等不同目标。接收器采用前后端分离的设计：

    • 前端 (Frontend)： 负责接收日志记录，并决定同步、异步等传递策略。

    • 后端 (Backend)： 负责执行实际的格式化操作并将日志写入最终目标。

寄存器名称与主要用途

寄存器 传统名称 主要用途
RAX 累加器 (Accumulator) 函数返回值、算术运算、系统调用号
RCX 计数器 (Counter) 第4个整数参数、循环计数、通用寄存器
RDX 数据寄存器 (Data) 第3个整数参数、与RAX配合用于乘除法、通用寄存器
RBX 基址寄存器 (Base) 被调用者保存、常用于存储基地址
RSP 栈指针 (Stack Pointer) 指向当前栈顶、永远指向内存中的栈顶
RBP 基址指针 (Base Pointer) 被调用者保存、通常用作栈帧基址（可选）
RSI 源索引 (Source Index) 第2个整数参数、字符串/数组操作源地址
RDI 目的索引 (Destination Index) 第1个整数参数、字符串/数组操作目的地址

Makefile

在Linux环境下，常常使用make程序自动化变异大型源码。
项目提供一个Makefile文件，make以此为基础进行程序的编译和链接。
make并不限定为构建C/C++项目服务，还可以用于任何文件的组织与处理，但主要用于Unix/Linux环境的自动化开发。

GNU make

Makefile基础

在Linux环境下，输入make命令，程序在当前目录下查找一个名为Makefile的文件，然后，根据该文件定的规则，自动化的执行任意命令。

规则

Makefile文件由若干条规则(Rule)构成，每一条规则指出一个目标文件(Target)，若干依赖文件(prerequisites)，以及生成目标文件的命令。

基础版本控制

该笔记操作设定在以下仓库中进行
mkdir test-dir

  1. 添加仓库
    使用 git init 进行仓库的初始化，此时文件夹 test-dir/ 已经绑定到一个 git Repo 。
    ./test-dir $: git init

  2. 将文件添加到版本库

    1. 使用 git add [filename] 将[filename]文件加入版本库。
      ./test-dir $: git add README.md
    2. 使用 git commit -m “describe” 将添加到版本库中的文件提交到仓库，其提交描述为 “describe” 。
      ./test-dir $: git commit -m "Add the README.md file into git Repo"
    • commit 相当于为仓库添加一个 版本快照 。
    • commit 对版本快照进行管理。
