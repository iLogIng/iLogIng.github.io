分离线程

使用 .detach() 在 boost::thread 对象上 进行线程分离，此时，该线程不再受主线程的限制，而是从主线程分离，又系统进行控制，其执行内容永远正确时可以使用。

try_join_for(boost::chrono::) 是等待 一段时间 若该时间内，子线程已经返回，则直接返回，若超出该设定时间，则子线程在超时时直接返回。设定了子线程的允许可运行时间。

try_join_until(boost::chrono::timepoint) 是等到 某个时间点 ，子线程直接返回。设定了子线程在何确定时刻会结束。

中断

可以调用 boost::thread 对象中的 interrupt() 成员函数来中断正在执行的线程。

与其他异常不同，当 boost::thread_interrupted 从线程主函数中传播出来时，这不会导致调用 std::terminate；效果就像线程主函数已正常返回一样。即，若不在主函数中使用异常捕获时，其并不抛出异常。

在函数对象中 调用 boost::this_thread::interruption_point() 来 标记中断点 。显示检查中断点。

在函数中可以通过 声明 boost::this_thread::disable_interruption disable; 来保证该线程 不被中断 。但是中断请求仍会被记录。

可以在函数中 调用 boost::this_thread::interruption_enabled() 来检查该线程是否允许被中断，当声明了 上一条的对象 obj 时，其返回值为 false 。

为了 反转 线程可中断状态，可以 声明 boost::this_thread::restore_interruption restore(disable); 来反转上一条 boost::this_thread__disable_interruption 对象的状态，让其后的语句进入 可中断状态 。通常会使用 { } 一个暂时块，来进行处理，在出块后 restore 被析构，线程状态回到原有的状态。当然，若 disable 也在一个暂时块中，线程的 不可中断 也是暂时的。

若在 disable 的作用时间内其 boost::thread 对象若使用了 interrupt() 方法，则其 中断信号 仍被记录，但是并不会立即中断线程。在后来若声明了 restore 则线程变为 可中断状态 若有 中断记录 则直接中断。

可以使用 boost::this_thread::interruption_requested() 来检查，该线程是否接收到过 中断信号 ，一般在 disable 的作用时间进行调用。因为，若在 restore 后，则有可能该线程早已中断。

线程ID

boost::thread::id 类的对象可用于标识线程。 每个正在运行的执行线程都有一个唯一的 ID，可以从相应的 boost::thread 通过调用 get_id() 成员函数获得，或者通过从线程内部调用 boost::this_thread::get_id() 获得。boost::thread::id 类的对象可以复制，并用作关联容器中的键：提供了全范围的比较运算符。 线程 ID 也可以使用流插入运算符写入输出流，但输出格式未指定。

boost::thread::id 的每个实例要么引用某个线程，要么引用 非线程 Not-any-thread。 引用 非线程 的实例彼此比较相等，但不等于引用实际执行线程的任何实例。 boost::thread::id 上的比较运算符为每个不相等的线程 ID 产生一个全序。

boost::thread::中的其他内容

boost::thread::hardware_concurrency() 该方法为静态方法，返回当前系统上可用的硬件线程数（例如，CPU 或核心或超线程单元的数量），如果此信息不可用，则为 0。

boost::thread::physical_concurrency() 该方法为静态方法，返回当前系统上可用的物理核心数。与 hardware_concurrency() 相比，它不返回虚拟核心的数量，而仅计数物理核心。

boost::thread::native_handle() 返回 native_handle_type 的实例，该实例可以与平台特定的 API 一起使用，以操作底层实现。如果不存在此类实例，则 native_handle() 和 native_handle_type 不存在。

可以对本线程信息进行操作。当然，对于入口函数 int main() 也具有操作能力。

yield() 函数，放弃当前线程剩余的时间片，以允许其他线程运行。可以避免一个线程过多的占用CPU的流水线时间。而非使用类似于 sleep_for() 的计时函数，占用CPU资源，让程序陷入一个忙等待状态，高强度占用CPU时间。也就是让该线程放弃对CPU的占用，释放这一部分的资源，让其他的线程 有机会竞争该时间片资源 。但，让出的操作，也会产生资源开销，最好不要频繁的调用该函数，只在关键优化部分进行调用。

yield() 仅仅是 让出时间片 ，可能在让出后，仍没有其他可用线程被CPU接收执行，此时CPU相对空闲，所以，该线程可能会直接回到CPU调度的时间片中执行。

  • 其主要的价值在于
    1. 减少CPU竞争：在自旋锁、CAS循环等高竞争场景中减少总线争用
    2. 提高系统响应性：让等待的线程主动让出CPU，使其他线程能更快响应
    3. 实现协作式调度：在计算密集型任务中穿插yield，提高整体吞吐量
    4. 节能：减少不必要的忙等待，降低CPU使用率

at_thread_exit(Callable func) 函数，传入一个函数对象func，func 的副本被放置在线程特定的存储中。当当前线程退出时，将调用此副本（即使线程已被中断）。也就是标识当前线程推出后需要执行的功能操作。

关于thread_group类

thread_group 提供了一个以某种方式相关的线程集合。可以使用 add_thread 和 create_thread 成员函数将新线程添加到组中。thread_group 不可复制或移动。

仅有一个空参的构造函数，创建为一个空的线程组集合。

create_thread(Func F) 在线程组中，创建一个线程，其传参形式与 new boost::thread(); 一致。

add_thread(boost::thread thrd)* 在线程组中，以对象指针的形式加入一个线程。获取由 thrd 指向的 boost::thread 对象的 所有权 ，并将其添加到组中。一般传入 new thread() ，或传入一个指向线程对象的指针。

remove_thread(boost::thread thrd)* 在线程组中将 thrd 移除，但并不调用 delete 。

size() 则是该线程组中所包含的线程数。

this_thread_in() 该线程组调用时所处的线程是否在线程组中。

is_this_thread_in(boost::thread thrd)* 线程指针thrd所指向的线程对象是否处于该线程组中。

内部锁定

在并发任务中，若存在多个读取与写入，由于线程竞争的问题，极可能会存在内存操作混乱以及其他不安全的状况。

此时，可以使用 互斥量mutex 来保证 独占访问 。互斥量是获取 同步 的简单而基本的机制。

在具有多个 并发对象 和多个 共享对象 的系统中，可能很难通过互斥量来描述同步。大量的互斥量的加入，可能会导致程序 难以阅读和编写 。所以将引入通用类来处理更复杂的同步和通信形式。

使用 RAII 的习惯，通过一个包装的 作用域锁 来简化很多编写需要。即 lock_guard 。

boost::mutex

lock() unlock() 两个成员方法，进行加锁的解锁，注意，在对一组并发对象与共享对象系统中的加锁与解锁需要的是 唯一一个 boost::mutex 对象 。如，以下的 test3() 函数中的线程传入的用于循环的Labmda函数，不在各个线程中创建各自的互斥量，而是使用 共享的互斥量 来处理线程的同步。

try_lock() try_lock_for() try_lock_until() timed_lock() 是尝试进行线程的加锁，分别是 尝试加锁，在一定时间内可否加锁，在什么时刻可以加锁，什么时间后加锁。

initialize() destory() 方法是暴露的对 boost::mutex 进行手动 初始化 销毁 的低级接口，可以用此控制 互斥量的生命周期 。构造时使用 initialize() 析构时调用 destory() 。

在 lock_guard 这个互斥量包装器中，其对作用域内的自动管理也是这样，lock_guard()构造时调用 mutex.lock() 其 ~lock_guard() 析构函数中调用 mutex::unlock() 。

关于 lock_guard 类，它暴露出来的方法除了 ~lock_guard() 析构函数 并没有其他可操作的方法，所以可能在某些情况下，会需要使用 { } 作用域 进行对 lock_guard 锁的作用域限定。

内部和外部锁定

内部与外部则是 锁 相对于 资源 的位置，在资源内部调用则为 内部锁 ，在资源外进行显示调用则为 外部锁 ，由 程序员进行处理 。

使用内部锁定的类保证对其公共成员函数的任何并发调用 都不会破坏该类的实例 。这通常通过让每个 公共成员函数 在 进入时获取对象的锁 来确保。 这样，对于该类的任何给定对象，在任何时候只能有一个成员函数调用处于活动状态，因此操作可以很好地序列化。（对同一字段的 写 读 操作的公共成员函数）

对于许多现实生活中的同步任务，内部锁定是不够的。如，若严格按 先增加 后减少 的顺序进行（如，在银行进行取钱及其扣除手续费的操作，两个操作之间不能出现其他操作）。

    • 支持两种锁定模型（内部和外部）
    • 高效；不使用不必要的锁定。
    • 安全；未经适当的锁定无法操作相应对象。

    1. 只要许可证仍然有效， object 对象就会 保持锁定状态
    2. 当 lock_guard 被销毁时， object 的 互斥锁将被释放 。

    最终，可以在代码中的任何部分， 只要 拥有一个 lock_guard对象 就能 保证一个 object 被锁定 。但是无法确切的知道是 哪个 count_test 对象被锁定

    此时，对 lock_guard类模板进行增强 。并将其增强后的版本称为 strict_lock 。本质上，strict_lock 的作用仅仅作为自动变量存在于 栈 上。 strict_lock 必须遵循非复制和非别名策略 。

    • 只能从一个 有效的 T 对象开始创建 boost::strict_lock 。

    • 可以通过 引用 将 boost::strict_lock 传递给函数或从函数返回。

    以上这些规则都是为了一个目的 —— 强制保证拥有一个 boost::strict_lock 意味着

    1. 锁定了一个 T 对象
    2. 该对象 稍后会被解锁

    lock_guard 实际上并不是直接锁定 BankAccount类对象，而是锁定了与这个BankAccount关联的互斥量，而关联是在外部维护的，从 类型系统上看 ，程序员只知道这个锁是用于 BankAccount类型 的资源，但不知道哪个具体实例。

互斥锁概念

互斥锁对象有助于 防止数据竞争 ，并允许线程之间的数据进行 线程安全同步 。线程通过调用锁函数之一 获得 互斥对象的 所有权 ，并通过调用相应的解锁函数 放弃 所有权 。
Boost.Thread 提供具有独占所有权语义的 递归 和 非递归 互斥锁，以及 共享所有权（多读、单写） 互斥锁。
以下的概念中都有集成有 CPP类型萃取 进行类型分析，但在笔记中并不详述。命名都是 **is_**格式

  • 互斥量的关键设计点

    它协调的是不同线程对同一资源的访问，而不是跟踪哪个具体线程应该访问。所有权机制确保同一时间只有一个线程可以进入临界区，无论这个线程是谁。

  • Boost.Thread 支持锁对象的四个基本概念：

    • Lockable
    • TimedLockable
    • SharedLockable
    • UpgradeLockable

    每种互斥锁类型都实现以上概念中的一个或多个，各种锁类型也是如此。

  • BasicLockable 概念

    <boost/thread/lockable_concepts.hpp>

    此概念，对 独占所有权进行建模 。如果以下表达式格式良好并具有指定语义，则该类型满足 BasicLockable 要求。

    通过 lock() 或的所有权必须通过 unlock() 释放

    • obj.lock();

      通过该调用 获得 锁的所有权

      要求 如果互斥锁不是递归的，则调用线程不拥有互斥锁。（调用线程：当前正在执行该函数的线程）即，不能在同一线程中连续重复获取互斥锁。

      效果 当前线程阻塞，直到可以为当前线程获得所有权。即，阻塞结束后才可能获取互斥锁。

      同步 对同一对象执行先前的 unlock() 操作与此操作同步。即，线程获取的锁的所有权在使用后需要主动释放。

    • obj.unlock();

      通过该调用 释放 锁的所有权

      要求 当前线程拥有 obj 。

      同步 此操作与后续获得同一对象所有权的锁操作同步。

      效果 释放 当前线程对 obj 的 锁定

  • Lockable 概念

    若类型满足 Lockable 的要求，则其也 必须满足 BasicLockable 的要求，并且以下表达式形势良好且具有指定的语义。

    • obj.try_lock()

      通过该调用获得锁的所有权，且必须通过调用 unlock() 释放

      要求 若互斥锁不是递归的，则调用线程不拥有互斥锁。

      效果 尝试 为当前线程获取所有权而 不阻塞 。即，仅仅是立即判断是否可以获取锁，若可，则获得锁；若非，则不获取锁，同时不阻塞，继续执行可用下文。

      同步 若 try_lock() 返回 true ，则对同一对象先前的 unlock() 操作与此操作同步。

      注意 由于 unlock() 不与随后失败的 try_lock() 同步，因此可见性规则非常弱，以至于在获取失败后几乎无法了解状态，即使在没有虚假失败的情况下也是如此。

  • TimedLockable 概念

    TimedLockable 概念细化了 Lockable 概念，以添加在尝试获取锁时对超时的支持。

    如果类型满足 TimedLockable 的要求，则它必须满足 Lockable 的要求，并且以下表达形式良好且具有指定的语义。

    变量

    • obj 为互斥量类型的值
    • rel_time 表示 chrono::duration 的实例
    • abs_time 表示 chrono::time_point 的实例

    表达式

    • obj.try_lock_for(rel_time)

      要求 锁互斥锁不是递归的，则调用线程不拥有互斥锁。

      效果 尝试为当前线程获取所有权。 阻塞 ，直到可以 获取所有权 或 超过了指定等待的时间 。

      同步 若返回 true 则对同一对象先前的 unlock() 操作同步。

    • obj.try_lock_until(abs_time)

      要求 若互斥量不是递归的，则调用线程不拥有互斥锁。

      效果 尝试为当前线程获取所有权。 阻塞 ，直到可以 获取所有权 或 达到指定时间 。

      同步 若返回值为 true 则对同一对象先前的 unlock() 操作同步。

    • obj.timed_lock(rel_time)

    • obj.timed_lock(abs_time)

    通过以上四个表达式获得的锁，必须通过调用 unlock() 进行释放

  • SharedLockable 概念

    之前的 lock() unlock() 等方法都是 独占锁 写者锁
    以下的 *_shared() 方法都是 共享锁 读者锁

    SharedLockable 概念 是 TimedLockable 概念 的改进， 允许共享所有权 和 独占所有权 。是标准的 多读/单写 模型：最多只能有一个线程拥有独占所有权，如果任何线程拥有 独占所有权 ，则其他线程 不能拥有共享或独占所有权 。或者，许多线程可以拥有共享所有权。

    如果类型满足 SharedLockable 的要求，则其必须满足 TimedLockable 的要求，并且以下表达式格式良好且具有指定的语义。

    变量

    • obj 表示类型的值
    • rel_time 表示 chrono::duration 的实例
    • abs_time 表示 chrono::time_point 的实例

    表达式

    • obj.lock_shared()

      效果 当前 线程阻塞 ，直到可以为当前线程获得共享所有权。

      后置条件 当前线程拥有 obj 的共享所有权。

    • obj.try_lock_shared()

      效果 尝试在 不阻塞 的情况下为当前线程获取共享所有权。

      后置条件 如果调用返回 true ，则当前线程拥有 obj 的共享所有权。

    • obj.try_lock_shared_for(rel_time)

      效果 尝试为当前线程获取共享所有权。 阻塞 直到获得共享所有权，或者经过指定的时间。如果指定的时间已经过去，则表现为 try_lock_shared() 。

    • obj.try_lock_shared_until(abs_time)

      效果 尝试为当前线程获取共享所有权。 阻塞 直到获得共享所有权，或者达到指定的时刻。若指定的时刻已经过去，则表现为 try_lock_shared() 。

    • obj.unlock_shared()

      先决条件 当前线程拥有 obj 的共享所有权。

      效果 释放当前线程对 obj 的共享所有权。

      后置条件 当前线程不再拥有 obj 的共享所有权。

    • obj.timed_lock_shared(abs_time)

      效果 尝试为当前线程获取共享所有权。 阻塞 直到获得共享所有权，或达到指定的时刻。如果指定的时刻已经过去，则表现为 try_lock_shared() 。

    通过调用以上方法获取锁的所有权，必须通过 obj.unlock_shared() 进行释放。

  • UpgradeLockable 概念

    UpgradeLockable 概念 是对 SharedLockable 概念的改进，允许 可升级所有权 以及 共享所有权 和 独占所有权 。

    单个线程可以拥有 可升级所有权 ，同时 其他线程可以拥有 共享所有权 。

    可升级所有权线程，可以随时尝试将其所有权升级为 独占所有权 。若没有其他线程拥有共享所有权，则升级立即完成，此时必须通过调用 unlock() 来释放锁。

    若在其他线程拥有共享所有权是进行升级，则尝试将失败，并且该线程将被 阻塞 ，直到可以获取 独占所有权 。

    所有权可以被 降级 或 升级 。

    共享所有权 → 可升级所有权 → 独占所有权

    类型满足 UpgradeLockable 要求，则它满足 SharedLockable 要求，并且以下表达式格式良好且具有指定的语义。

    通过调用 lock_upgrade() 获取的锁所有权必须通过调用 unlock_upgrade() 来释放。如果通过调用 unlock_xxx_and_lock_yyy() 函数更改了所有权类型，则必须通过相应的所有权释放函数进行所有权释放。

    • obj.lock_upgrade()

      先决条件 调用线程不拥有互斥锁

      效果 当前线程 阻塞 ，直到可以为当前线程获得升级所有权。

    • obj.unlock_upgrade()

      先决条件 当前线程拥有 obj 的升级所有权。

      效果 释放当前线程对 obj 的升级所有权

      同步 此操作与后续获得同一对象所有权的锁操作同步。

条件变量

<boost/thread/condition_variable.hpp>

类 condition_variable 和 condition_variable_any 提供了一种机制，允许一个线程 等待 另一个线程通知某个特定条件已变为真。

一般用法是，一个线程锁定一个互斥量，然后在 condition_variable 或 condition_variable_any 的实例上调用 wait 。当线程从 等待中唤醒 时，它会检查相应的条件是否现在为真，若是，则继续。若非，则调用 wait() 继续等待。有两种返回值，bool cv_stat枚举量。

注意，lock被传递给wait，wait原子的将线程添加到等待条件变量的线程集合中，并解锁互斥锁。当线程被唤醒时，互斥锁将在调用wait返回之前再次被锁定。这允许其他线程获取互斥锁以更新共享数据，并确保与条件相关联的数据被正确同步。

同时，另一个线程将条件设置为 true ，然后在条件变量上调用 notify_one 或 notify_all 以分别唤醒（通知）一个等待线程或所有等待线程。

同步原语 wait 与 线程操作 yeild 的区别在于：
yeild 是由线程自身让出时间片给其他线程。是一种 就绪状态 。仍消耗CPU周期。
wait 是为了等待特定的同步而主动进入阻塞状态。是一种 等待/阻塞状态 。CPU周期使用为零。

条件变量的同步原语 wait 中可以传入一个返回值为 bool 的谓词。
注意，在更新共享数据之前，会锁定一个互斥锁，但互斥锁不必再调用 notify_one 时保持锁定。

单次初始化

<boost/thread/once.hpp>

boost::call_once 提供了一种机制，用于确保初始化例程只运行一次，而不会发生数据竞争或死锁。

boost::call_once(once_flag& flag, Function& f, ArgTypes&&… args)

要求 Function和每个ArgTypes都是 可移动构造的 ，并且 invoke(decay_copy(boost::forward(f)), decay_copy(boost::forward(args))…)应该格式良好。

效果 对同一个 once_flag 对象调用 call_once 是串行化的。如果在同一个once_flag 对象上，之前并没有有效的 call_once 调用，则会像调用invoke(decay_copy(boost::forward<Function>(f)), decay_copy(boost::forward<ArgTypes>(args))...) 一样调用参数 func，并且当且仅当 invoke(decay_copy(boost::forward<Function>(f)), decay_copy(boost::forward<ArgTypes>(args))...) 返回且没有抛出异常时，call_once 的调用才生效。如果抛出异常，则该异常会传播给调用者。如果之前在同一个 once_flag 对象上存在有效的 call_once 调用，则 call_once 会返回而不调用 func。

屏障（barrier）

<boost/thread/barrier.hpp>

屏障是一个简单的概念，也被称为 汇合点 ，它是 多个线程之间的同步点 。

    效果 构造一个用于 count 个线程的屏障

  • 构造函数 barrier(unsigned int count, F&& completion)

    要求 完成函数调用 completion() 的结果类型是 void 或 unsigned int 。

    效果 构造一个用于 count 个线程和一个完成函数 completion 的屏障。

  • 析构函数 ~barrier()

    先决条件 没有线程在 *this 上等待。

    效果 销毁*this

  • bool wait()

    效果 阻塞 ，直到 count 个线程在 *this 上调用 wait 或 count_down_and_wait 。当第 count 个线程调用 wait 时，屏障会被 重置 ，并且所有等待线程都会被 解除阻塞 。

    重置取决于屏障是否使用 完成函数构造 。如果没有完成函数或者完成函数的结果为 void ，则重置包括恢复原始计数。否则，重置包括分配完成函数的结果（必须不为0）。

    返回值 对于每批等待线程中恰好的（就是最后一个wait的线程，直接导致屏障重置的线程）一个线程为 true ，否则为 false 。

    在上述用例中， barrier构造函数传入的 count 为 3，而创建了 N=5 个线程，在进入各个线程后直接使用 bary.wait() 进入barrier锁定，根据线程的启动顺序，可见，仅仅会（极大概率）输出 0，1，2 三个线程分配的 i 值。因为是这三个线程最先开始。这三个线程进行了同步。

    当然，其中由于 N=5 count=3 最后只剩下 2 个线程，程序一直处于等待中。（ 屏障数 与 线程数 不匹配 时，会导致 死锁 ）

    wait 返回值规律：每批到达屏障的线程中， 恰好有一个线程 会受到 true 其他收到 false

    屏障的核心价值在于：
    同步点 确保多个线程在特定点同步
    阶段控制 协调多阶段并行算法的执行
    数据一致性 在关键操作前确保所有线程就绪

    屏障重置的三种情况

    其中，返回为 unsigned int 的完成函数，可以支持 动态的 调整并行策略。

  • void count_down_and_wait()

    效果 阻塞 ，直到 count 个线程在 *this 上调用 wait 或 count_down_and_wait 。当第 count 个线程调用 wait 时，屏障会被重置，并且所有等待线程都会被 解除阻塞 。

    重置取决于 屏障是否使用完成构造函数 。如果 没有完成函数 或 完成函数的结果是 void ，则重置包括恢复原始计数。否则，重置包括分配完成函数的结果（必须不为0）

    count_down_and_wait() 功能与 wait() 的功能完全相同，而是只需要同步，而不用像 wait() 一样，不用关心哪个线程触发完成函数。（即，不关心 领导线程 。）

锁存器 latch

锁存器是一种 线程协调机制 ，允许一个或多个线程阻塞，直到一个或多个线程到达某个点。

latch

latch 维护一个 内部计数器 ，该计数器在 创建latch时初始化 。一个或多个线程可能会 阻塞等待 ，直到计数器递减到 0 。

latch 的实例不可复制或移动。

  • 构造函数 latch(std::size_t count)

    效果 构造一个 latch ，其中包含内部计数器的 初始值 。

    注意 计数器 可以为零

  • 析构函数 ~latch()

    先决条件 没有线程正在等待或调用 *this 上的 count_down 。

    效果 销毁 *this latch

  • wait()

    效果 阻塞调用线程，直到内部计数 达到零值 。然后，所有等待线程都被 解除阻塞 。

  • try_wait()

    返回值 如果内部计数为 0，则返回 true，否则返回 false。 不会阻塞调用线程 。

  • cv_status wait_for(const chrono_duration<Rep, Period>& rel_time)

    效果 阻塞调用线程 ，直到内部计数器达到零值或经过 rel_time 时间。如果 没有超时 ，所有等待线程都会被解除阻塞。

    即，等待一段时间内确定是否达到计数器耗尽。

    返回值 cv_status类型 若内部计数为0则 cv_status::no_timeout ，若经过 rel_time 时间，则为 cv_status::timeout 。

  • cv_status wait_until(const chrono::time_point<Clock, Duration>& abs_time)

    类似于之前所有的 until 成员定义。

  • count_down()

    要求 内部计数器非零。

    效果 将内部计数 -1 ，然后返回。若 计数器达到0 ，则将 释放 wait() 中阻塞的任何线程。

  • count_down_and_wait()

    结合调用了 count_down() 与 wait()

  • reset(size_t)

    要求 只有在当前没有其他线程在等待函数中时才能调用此函数。

    返回值 使用新的初始线程计数值 重置 latch

boost::barrier 线程间相互等待。可重复使用。
boost::latch 单向等待。一次性重复。（可在用尽后重置）

Futures

futures 库提供了一种处理同步 future 值的方法，无论这些值是由另一个线程生成，还是在单个线程上相应外部刺激，或按需生成。

boost::promise boost::futures boost::packaged_task 是对上述同步原语的更高级别的抽象。用于简化并发编程，特别是异步操作和线程间传递结果。

通过以下四个类模板来实现：

  • future
  • boost::shared_future 用于检索异步结果
  • boost::promise
  • boost::packaged_task 用于生成异步结果

future 的一个实例拥有 对结果的唯一引用 。

所有权可以在实例之间使用 移动构造函数 或 移动赋值运算符 进行转移，但最多只有一个实例拥有对 给定异步结果的引用 。当结果准备好时，它会通过 右值引用 从 boost::future::get() 返回，以便可以根据类型适当地 移动 或 复制 结果。

另一方面， boost::shared_future 的多个实例 可能引用相同的结果 。实例可以自由复制和分配，并且 boost::shared_future::get() 返回 const 引用，以便可以安全地多次调用 boost::shared_future::get() 。可以将 future 的实例移动到 boost::shared_future 的实例中，从而转移关联的异步结果的 多有权 ，但反之则不可。

boost::async 是一种运行异步任务的简单方法。调用 boost::aync 会返回一个 future ，该future将包含任务的结果。

可以单独使用future，也可以使用 boost::wait_for_any() 和 boost::wait_for_all() 函数之一等待。

  • 创建异步值

    可以使用 boost::promise 或 boost::packaged_task 设置 future 中的值。

    boost::packaged_task 是一个可调用对象，它包装一个函数或可调用对象。调用 packaged_task 时，它会 依次调用包含的函数 ，并使用返回值填充 future 。

    boost::promise 稍微低级一些：它只提供显式函数来将值或异常存储在关联的future中。因此，当值可能 来自多个可能的源 ，或者 单个操作可能产生多个值 时，可以使用promise。

  • 等待回调和延迟 Future

    boost::promise 和 boost::packaged_task 都支持 等待回调 ，当线程在对 wait() 或 timed_wait() 的调用中 阻塞时 ，会调用该回调，该回调发生在等待来自 boost::promise 或 boost::packaged_task 的结果的future上时，该回调在执行等待的线程中。可以使用相应的 boost::promise 或 boost::package_task 上的 set_wait_callback() 成员函数来设置这些回调。

    这允许 延迟future ，其中直到某些线程需要结果时才实际计算结果。

    上述示例中， f.get() 的调用会回调 invoke_lazy_task ，该回调运行任务以设置该值。如果删除对 f.get() 的调用，则永远不会运行该任务。

  • 处理分离线程和线程特定变量

    分离线程给具有线程存储持续时间的对象带来了问题。如果使用除了 boost::thread::join() 的机制来等待 线程完成工作，如，等待future准备就绪，那么线程特定变量的析构函数 仍然会在等待线程恢复后运行 。方法是 确保在future 准备就绪之前 销毁具有线程存储持续时间的对象。

  • 异步执行

    boost::async 提供了一种简单的方式来运行异步任务，以利用可用的硬件并发性。

    调用 boost::async 会返回一个 boost::future ，该future将包含任务的结果。根据启动策略，任务可以在其自身的线程上异步运行，也可以在调用该future的wait() 或 get()成员函数的线程上同步运行。

    启动策略可以是 boost::launch::async（要求运行时创建一个异步线程） 或 boost::launch::deferred（表示只想函数调用延迟到稍后的时间，即惰性求值） 。此参数是可选的，若省略，函数将使用默认策略。

future类模板

  • 构造函数

    • 默认构造 future()

      效果 构造一个未初始化的 future

      后置条件 this→is_ready() return false ，this→get_state() return boost::future_state::uninitialized 。

    • 移动构造

      效果 构造一个 新的 future，并将与other关联的共享状态的 所有权转移 到 *this

      后置条件 this→get_state() 返回调用前 other→state() 的值。other→get_state() return boost::future_state::uninitialized 。如果 other 与共享状态 相关联 ，则该结果现在与 *this 相关联。other在执行后 不与任何共享状态相关联 。

  • 析构函数

    • ~future()

      效果 销毁 *this

  • get()

    效果 如果*this与共享状态关联，则等待结果准备就绪，如同调用 boost::future::wait() 一样，并检索结果。

    返回值 R&存储值的引用 &&右值引用 void无返回值。

    注意 get() 是一个中断点。

  • wait()

    效果 如果*this与共享状态关联，则 等待 直到结果准备就绪。如果在进入时结果 尚未准备就绪，并且结果 设置了等待回调 ，则在等待之前调用该回调。即，先回调再等待。

    后置条件 this→is_ready() return true 。this→get_state() return boost::future_state::ready

    注意 wait() 是个中断点。

  • wait_for(chrono::duration)

    效果 *this与共享状态关联，则等待直到结果准备就绪，或经过 指定的时间 。如果再进入时结果未准备好，并且结果设置了等待回调，则再等待之前会调用该回调。

    返回值 future_status::deferred ready timeout

  • wait_until(chrono::time_point)

    效果 *this与状态关联，则等待结果准备就绪，或者抵达 指定的时刻 。如果再进入时结果未准备好，并且结果设置了等待回调，则在等待之前会调用该回调。

  • valid()

    返回值 *this与共享状态关联，则为true否则为false

    备注 此函数的结果不稳定，即使该函数返回true或反之亦然，future也可能变为无效。

  • is_ready()

    返回值 如果*this与共享状态关联，并且该结果 已经准备好进行检索 ，则为true，否则false

    备注 此函数的结果不稳定，即使该函数返回true或反之亦然，future也可能变为未就绪。

  • has_value()

    返回值 如果*this与共享状态关联，并且该结果是 存储的值 ，则为true，否则false。

    备注 此函数的结果不稳定，即使该函数返回true或反之亦然，future也可能 丢失其值 。

  • has_exception()

    返回值 如果this与共享状态关联，并且该结果 是*存储的异常 ，则为true，否则false。

    备注 此函数的结果不稳定，即使该函数返回true或反之亦然，future也可能 丢失其异常 。

  • get_state()

    效果 确定与*this关联的共享状态的状态。

    返回值类型 boost::future_state

  • share()

    返回值 shared_future(boost::move(*this)) 即，可以让 unique_future类型 由此 变为 shared_future 。

    后置条件 this→valid() == false

shared_future 类模板

包含所有上述 future类模板 中的成员函数。

shared_future的 get() 可以调用多次（而非一次），可由多个线程获取（而非一次性），可以自由 复制 和 赋值 unique_future仅支持移动构造和移动赋值。

若需要在 多线程的环境 中使用，对同一个异步结果进行访问，则 shared_future 是唯一安全的选择，因为它是 线程安全的 。

promise 类模板

  • 构造函数

    • 默认构造函数 promise()

      效果 构造一个新的 boost::promise 且不关联任何结果。

    • 移动构造

      效果 构造一个新的 boost::promise 并将其与other关联的结果的所有权转移到*this，使other不再关联任何结果。

  • 析构函数

    效果 销毁this。如果有一个与this关联的结果，并且该结果 尚未准备就绪 ，则将该任务关联的任何future设置为 准备就绪 ，并将 boost::broken_promise 异常作为结果。

  • get_future

    效果 如果this没有与结果关联，则为新的共享状态分配存储空间并将其与this关联。返回一个与*this关联的 结果相关联 的future。

    效果 任何 阻塞 等待异步结果的线程都会被唤醒。即，ready状态。
