Asio

Boost.Asio 是一个跨平台的C++库，用于网络和底层I/O编程，它使用现代C++方法为开发人员提供一致的异步模型。

定时器

通过展示如何对定时器执行阻塞来介绍 asio
所有的 asio 类都可以通过简单的包含 <boost/asio.hpp> 头文件来使用

Timer 1

所有使用 asio 的程序都需要 至少有一个 I/O 执行上下文 ，例如 io_contex 或 thread_pool 对象。I/O 执行上下文提供对 I/O 功能的访问。

我们首先在函数的开头声明一个 io_context 类型的对象。

接下来的 boost::asio::steady_timer 类型的对象。提供 I/O 功能的和兴 asio 类总是接受一个执行器，或者对执行上下文的引用（如 io_context），作为它们的第一个构造函数参数。构造函数的第二个参数将定时器设置为从现在起 2700 毫秒后到期。

以上示例中，对定时器执行 阻塞等待 。即，对 steady_timer::wait() 的调用将不会返回，直到定时器过期。定时器始终处于两种状态之一： 已过期 或 未过期 。如果在已过期的定时器上调用 steady_timer::wait() 函数，他将立即返回。

Timer 2

使用 asio 的异步功能意味着提供一个 完成令牌 。它决定了当 异步操作 完成时，结果将如何传递给 完成处理程序 。

调用 steady_timer::async_wait() 函数来执行异步等待。在调用函数时，我们传递上面定义的 print 函数。

最后，必须在 io_context 对象上调用 boost::asio::io_context::run() 成员函数。asio 库保证完成处理程序只会被当前正在调用 boost::asio::io_context::run() 的线程调用。因此，除非调用 boost::asio::io_context::run() 函数，否则异步等待完成的完成处理程序将 永远不会 被调用。

boost::aiso::io_context::run() 函数也会在还有 任务 时继续运行。

在本例中，任务是对定时器的异步等待，因此该调用将不会返回，直到定时器过期并且完成处理程序已返回。

io.run() 实际上是启动该异步上下文对象的事件循环。

Timer 3

要使用 asio 实现重复定时器，需要在完成处理程序中更改的定时器的到期时间，然后启动新的异步等待。完成处理程序需要能够访问定时器对象。为此可以向 print 函数添加两个新参数。

  • 一个指向定时器对象的指针
  • 一个计数器，以便在合适时停止程序

print_v2 中，没有显示调用来要求 io_context 停止。其中（io.run()的返回值为 定时器被触发的次数 ）关于 t→expires_at(t→expiry() + boost::asio::chrono::seconds(1)) 的调用，是调整定时器的到期时间前移一秒，通过相对于旧的到期时间计算新的到期时间。

然后启动一个新的异步等待定时器。 std::bind函数用于将额外的参数与完成处理程序关联。 steady_timer::async_wait() 函数期望一个具有 void(const boost::siystem::error_code&) 签名的处理函数。

绑定额外的参数将会将 print_v2 函数 转换 为与签名正确匹配的函数对象。

本例中， std::bind 的 boost::asio::placeholders::error 参数是传递给处理程序的错误对象的命名占位符。当启动异步操作时，如果使用 std::bind 必须仅指定与处理程序的参数列表匹配的参数。

Timer 5

可以使用 strand 类模板在多线程程序中同步完成处理程序。

asio库保证完成处理程序仅会从当前正在调用 boost::asio::io_context::run() 的线程中调用。因此，仅从一个线程调用 boost::asio::io_context::run() 可以确保完成处理程序不会并发运行。

单线程方法通常是开始开发使用 asio 的应用程序的最佳选择。缺点是它对程序，特别是服务器的限制：

  • 当处理程序可能需要很长时间才能完成时，响应性差
  • 无法在多处理器系统上扩展。

为了突破这些限制，可以使用线程池进行 boost::asio::io_context::run() 的调用。但是，由于允许处理程序并发执行，当处理程序可能访问共享的线程不安全资源时，需要一种同步方法。

其中 strand 类模板时一个执行器适配器，它保证对于通过它调度的处理程序，一个正在执行的处理程序将被允许在下一个处理程序开始前完成。无论有多少线程正在调用 boost::asio::io_context::run() ，这都得到保证。

处理程序仍然可以与未通过 strand 调度的其他处理程序，或通过不同的 strand 对象调度的处理程序并发执行。

当启动异步操作时，每个完成处理程序都 “绑定” 到一个 boost::asio::strandboost::asio::io_context::executor_type 对象。 boost::asio::bind_ex返回一个新的处理程序，该处理程序会自动通过 strand 对下给你调度其包含的处理程序。通过将处理程序绑定到同一个 strand 可以确保不会 并发 执行。ecutor() 函数

在多线程程序中，如果异步操作的处理程序访问共享资源，则应同步这些处理程序。（本程序中，print1 print2 使用的共享资源是 std::count 和 count_ 数组成员。

以上的用例中，在子线程与主线程分别调用了 boost::asio::io_context::run() ，此时，主线程并不会立即退出，而是会等待异步操作的完成。

在此例中，我们知道主线程的 io.run() 会阻塞，所以在程序线程的最后才会输出 “Main Thread io.run() is worked.” 的提示。

本例中，两次 io.run() 的调用：

  1. 两个线程 共享 同一个 boost::asio::io_context 对象
  2. 两个线程 共同处理所有的异步操作
  3. boost::strand 的作用是确保 线程安全

  • boost::asio::io_context

    执行器使用方法，代替了直接使用 boost::io_context引用 的模式，可以安全的从任何线程向一个上下文安全的投递任务。

    执行器（Executor）抽象了如何调度函数对象（任务事件）执行的机制。允许程序的任何位置、任何执行周期中，安全的向其所属的异步上下文注册事件处理程序。（是一个轻量级的“注册令牌”）廉价的复制对象、不拥有资源，上下文无关，时间无关，类型安全，生命周期安全。

    执行器是io_context的轻量级代理，它提供了一种类型安全、线程安全且与生命周期解耦的方式，允许在程序的任何执行阶段和任何执行上下文中，向其所属的异步事件循环注册事件处理程序。
