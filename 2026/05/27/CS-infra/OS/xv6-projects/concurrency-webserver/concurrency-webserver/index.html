原 wserver 程序处理过程

  1. 监听并读取客户端套接字 accept_or_die

  2. 处理客户端连接请求 request_handle

    1. 读取并解析客户端请求，分析请求行

    2. 根据请求类型 is_static 进行静态/动态请求的响应

      • 静态请求request_serve_static：调用mmap建立文件内存映射，读取本地文件，返回响应

      • 动态请求request_serve_dynamic：通过调起子进程执行CGI程序处理

  3. 处理完成，关闭连接 close_or_die

初始环境可能的问题

wserver.c 中的 optarg 未定义问题

  • 需要包含头文件 <unistd.h> 或 <getopt.h>
    • 头文件中已自然声明
      • *char optarg 指向当前参数
      • int optind 下一次调用 getopt() 时的下一个argv指针的索引
      • int opterr 标志，设置为0，不会向 stderr 输出错误信息
      • int optopt 最后一个未知选项

带参数的选项要在传入 getopt() 函数的字符串中添加 : 字符

  • 如 “A:B:C:” 则标识了 -A -B -C 三种参数
  • 使用 :: 标识可选参数
  • 必须检查 optarg 字段是否为空
  • getopt 默认会将选项参数进行重排，将非选项参数放至最后

request.c 文件的编译问题

  • 会遇到 sprintf 函数，缓冲区溢出问题
    • 可通过将 sprintf 更换为 snprintf 函数来解决该问题，需要传入该缓冲区的实际大小，根据 snprintf 的返回值判断是否产生了缓冲区截断。
    • 也可以使用 asprintf 函数进行缓冲区的自动扩展

设计

生产者-消费者 划分

生产者

  • 通过读取请求，写入缓冲
  • 在主线程使用 buffer_push 将 conn_fd 写入缓冲

消费者

  • 通过从缓冲中读取请求，写回响应
  • 在子线程中使用 buffer_pop 将 conn_fd 从缓冲中读出
