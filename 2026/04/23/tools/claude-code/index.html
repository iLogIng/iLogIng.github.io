  • env.ANTHROPIC_BASE_URL: 国内大模型的API端点URL

    • endpoint: API端点URL
    • api_key: API密钥

提示词工程

核心公式： 好的提示词 = 明确的角色 × 充分的上下文 × 结构化的输出 × 可验证的标准

提示词不是一次写成的，需要根据 AI 的回复不断修正迭代。

事先声明

  1. 将AI视为 协作者 而非个人的替身

    • 要保证项目代码的最终决策权在我们自己，你需要不断引导它分析约束、评估风险、给出可选方案，而不是直接接受第一个答案。

  2. 好的架构是分视图、分阶段、分模块的

    • 你必须在提示词里显式要求 AI 从不同架构视图（逻辑、过程、开发、部署、数据）思考，避免它过早陷入细节。
    • 你最好能事先规范化项目的顶层设计，再与AI进行总体的设计讨论，逐步深入。

  3. 每一次讨论都是一次 架构评审

    • 把提示词设计成微小的评审会议：输入上下文、假设，要求 AI 输出分析、质疑、建议。
    • 但是，绝不能被AI牵着鼻子走，在整理AI的建议时需要自己的思考，纠正AI偏离预期设计的想法。

CLAUDE.md 与对话 Prompt 的分工

存储位置 适合的内容 特点
CLAUDE.md 项目架构、编码规范、测试规范、目录结构、常用命令 持久化，自动加载到每次对话
对话 Prompt 当前任务上下文、具体需求、临时约束、输出格式 会话级，一次性，灵活调整

如何表述清楚任务

需要以 架构师 的方式进行思考
聚焦于”为什么做”、”如何做”
使用层进式的方式进行讨论
提出需求 -> 暴露问题 -> 修改架构

1. 上下文构建与约束澄清

目标： 让 AI 快速理解项目全貌，产出需求疑点清单和风险假设。

需要包含：

  1. 项目背景
    • 团队人员信息
    • 项目摘要
    • 项目周期
  2. 项目目标
    • 目标领域
    • 核心业务目标
      • 多个子目标…
  3. 项目关键约束

产出一份结构化的项目分析清单
包含：

  • 结构化的需求疑点清单
  • 风险假设列表

可复用 Prompt 模板：

你是一名资深[架构师/领域专家]，请以”评审者”视角分析以下项目信息：

项目背景

  • 项目名称：[项目名]
  • 团队规模：[人数]，角色：[前端/后端/测试/PM]
  • 项目周期：[开始] ~ [结束]
  • 当前阶段：[需求/设计/编码/测试]

核心目标

  • 业务目标：[一句话]
  • 关键指标：[如 QPS、响应时间、用户量]

已知约束

  • 技术栈：[语言/框架/数据库]
  • 继承系统：[如有，描述]
  • 合规要求：[如有，指明]
  • 资源限制：[预算/人力/时间]

请输出：

  1. 一份结构化的需求疑点清单（至少5条）
  2. 一份风险假设列表（按影响程度排序）
  3. 针对疑点给出的补充建议

常见反模式：

反模式 问题 正确做法
一次性堆砌所有信息 AI 注意力被稀释，关键约束被忽略 按层进式逐步给出，每一轮聚焦一个层面
只给目标不给约束 AI 可能选一个无法落地的方案 显式声明否定约束（”不使用XX”）
模糊指标 无法做验收 给出具体数字和范围

自检清单：

  • 角色定义是否具体？（”10年后端经验的Java架构师” vs “一名开发者”）
  • 是否给出了具体数字和范围？（避免”大量用户”、”很快”）
  • 是否说明了”不要做什么”？（否定约束同样重要）
  • 是否指定了输出格式？

2. 架构风格与关键决策

目标： 对比多个候选方案，输出决策记录（ADR）草稿。

  1. 项目的基础架构
  2. 各个待选的项目架构与现有约束的冲突
  3. 团队的组织能力需要

输出格式：
针对每个待选架构的风格简述，进行表格化输出，给出推荐理由
针对各个待选的架构风格，简述其与现有约束需要的冲突，以及如何权衡冲突
为最终的选择编写一份 Architecture Decision Record (ADR) 草稿

可复用 Prompt 模板：

[粘贴阶段一的产出]

请从可选架构方案中，评估以下候选项：

  1. [方案A：如微服务]
  2. [方案B：如单体+模块化]
  3. [方案C：如事件驱动]

评估维度：

  • 与现有约束的匹配程度（低/中/高）
  • 团队能力要求
  • 迁移成本
  • 长期可维护性

输出格式：

  1. 对比表格（方案 × 评估维度）
  2. 每个方案的权衡分析（2-3句话）
  3. 你的推荐方案 + 核心理由
  4. 一份 ADR（Architecture Decision Record）草稿

常见反模式：

反模式 问题 正确做法
直接让 AI 选方案 失去决策权，难以追溯原因 要求给出多个选项和权衡分析
忽略团队能力 选了最优方案但团队无法交付 把团队技术栈作为硬约束

自检清单：

  • 我是否要求了”对比”而非”单一推荐”？
  • 我是否给了明确的评估维度？
  • 我是否考虑了团队能力约束？

3. 结构设计与边界定义

目标： 产出模块划分、接口契约和依赖关系。

该部分用于：

  • 划分模块/服务
  • 定义边界上下文（DDD）
  • 确定各组件职责和接口契约

该部分是针对各个模块进行设计

产出格式：

  • 模块列表
  • 接口概要
  • 初步的模块与服务依赖图（使用mermaid文本输出）

可复用 Prompt 模板：

我们已确定采用[架构方案]，请以此为基础进行结构设计。

设计输入：

  • 核心业务用例：[列表]
  • 关键实体：[列表]
  • 跨切面关注点：[日志/鉴权/监控…]

设计原则：

  • [如：领域驱动、按业务能力划分]
  • [如：模块间松耦合、接口优先]

输出格式：

  1. 模块列表（名称 + 职责 + 依赖）
  2. 接口概要（模块间关键 API）
  3. 依赖关系图（使用 mermaid）
  4. 关键设计决策说明

常见反模式：

反模式 问题 正确做法
描述模糊（”设计一个系统”） AI 产出过于泛化 给出明确的模块划分依据和原则
不分层直接出细节 过早陷入代码级实现 先逻辑视图再过程视图，逐步聚焦

自检清单：

  • 每个模块的职责是否单一、无歧义？
  • 接口契约是否包含入参/出参/异常？
  • 是否明确了模块间的依赖方向？

4. 技术栈与实现原语定义

目标： 确定具体选型和编码规范。

该部分用于：

  • 深究框架、库、中间件的选择
  • 定义编码原则
  • 错误处理模式
  • 安全策略
  • 等实现原语

产出格式：

  • 选型决策表
  • 项目实现规范草稿

可复用 Prompt 模板：

请为以下模块做出技术选型：[模块列表]

选型约束：

  • 语言：[强制约束]
  • 许可协议：[限制性要求]
  • 团队熟悉度：[现有技术栈]
  • 性能要求：[具体指标]

请输出：

  1. 每个候选技术的对比表（框架 × [性能/社区/学习成本/许可]）
  2. 推荐方案及理由
  3. 编码规范草稿（命名、错误处理、测试策略、安全实践）

常见反模式：

反模式 问题 正确做法
只看技术先进不考虑生态 选冷门库后期维护困难 把社区活跃度和许可证纳入评估
忽视非功能性约束 上线后出现性能/安全问题 显式声明性能、安全、合规要求

自检清单：

  • 选型标准是否可量化？
  • 是否考虑了长期维护成本？
  • 编码规范是否包含”禁止模式”？

5. 评估、验证与文档生成

目标： 验证设计质量，生成可交付物。

该部分用于：

  • 使用AI对现有设计进行评审，
  • 使用AI将设计转化为相应的需求文档

可复用 Prompt 模板：

请以[特定角色，如安全工程师/性能专家]视角，评审以下设计：

设计输入： [粘贴阶段二~四的产出]

评审重点：

  1. [如：安全漏洞分析]
  2. [如：性能瓶颈识别]
  3. [如：边界情况覆盖]

输出格式：

  1. 发现的问题列表（严重程度 × 影响范围 × 修改建议）
  2. 未覆盖的场景分析
  3. 改进优先级排序

问题确认后，请根据最终设计生成：

  1. 技术方案文档
  2. API 接口文档
  3. [其他所需的文档类型]

常见反模式：

反模式 问题 正确做法
只让 AI “检查一下” 输出过于笼统无法落地 给出具体检查点（安全/性能/边界）
不指定受众 文档风格与使用者不匹配 说明文档受众（技术团队/管理层）

自检清单：

  • 评审角色是否匹配当前关注点？
  • 是否要求了优先级排序？
  • 文档受众是谁？
