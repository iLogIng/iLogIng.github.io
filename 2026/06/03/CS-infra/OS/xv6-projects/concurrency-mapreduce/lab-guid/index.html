MapReduce 实验指导

实验分析

本项目是 OSTEP（Operating Systems: Three Easy Pieces） 的并发 MapReduce 实验。目标是实现一个单机多线程简化版 MapReduce 框架，核心是实现 mapreduce.h 中声明的函数并设计线程安全的中心数据结构。

2. 设计核心数据结构（最关键的难点）

需要一个线程安全的数据结构来承接 MR_Emit() 发射的键/值对，并能被 reducer 线程按分区、按键排序地读取。

实际采用方案：分区内按 key 聚合 + 延迟分配

采用三层嵌套结构：

关键特性：

  • 分区粒度锁：每个 MR_partition 持有独立的 pthread_mutex_t，不同分区的并发写入天然无竞争。
  • 键级锁：每个 MR_key_values 内部持有独立的 pthread_mutex_t，保护 values_iter 迭代器，保证 reducer 线程安全消费。
  • 写入时去重：MR_Emit 阶段即完成 key 去重——同一 key 的多次 emit 会将 value 追加到同一个 MR_key_values.values[] 中，无需 map 后额外合并。
  • 延迟分配：kvs_groups 数组预分配骨架（raw memory），仅在 key 首次出现时才调用 MR_key_values_init 初始化对应 slot。未使用的 slot 仅占裸内存，无资源泄漏。
  • 深拷贝：框架对 key 和 value 均进行 malloc + strcpy，不依赖用户内存生命周期。

4. 并发控制要点

两级锁设计

  • 分区级锁（MR_partition.mutex）：保护 kvs_groups 数组的结构变更（添加新 key、扩容）。多个 mapper 线程并发写入不同分区时天然无竞争；写入同一分区时互斥。
  • 键级锁（MR_key_values.mutex）：保护 values_iter 迭代器。reduce 阶段 reducer 线程迭代消费 value 时加锁。

同步点

  • Map → Sort：thread_pool_destroy(map_pool) 内部 pthread_join 等待所有 mapper 完成后，才开始排序。
  • Sort → Reduce：排序在单线程中完成（MR_Run 主线程），之后才创建 reduce 任务。
  • Reduce 只读：reduce 阶段不再有写入，MR_partition_has_key() 无锁查找是安全的。

为什么不在 map 阶段就开始排序（流水线化）

第一版不做流水线是正确性优先的选择。map 全部完成再统一排序和 reduce，避免了”排序中又有新 key 插入”的复杂同步问题。如果需要性能优化，可以在后续版本中引入条件变量做流水线。

5. 内存管理要点

  • 深拷贝：MR_Emit 传入的 key/value 可能来自用户的栈或短期堆内存（如 strsep 返回的指针），框架必须在 MR_key_values_init 和 MR_key_values_push_value 中进行 malloc + strcpy。
  • 释放链：释放顺序与创建顺序相反。MR_key_values_destroy 释放 key 和 values 的内部内存，但不释放 kvs 结构体本身——它由 partition->kvs_groups 数组统一 free。
  • 延迟分配：MR_partition_init 仅分配数组骨架，不初始化内部 slot。未使用的 slot 不会泄漏 mutex 或 values 数组。扩容时 (realloc) 新增的 slot 同样是 raw memory，在取用时才 MR_key_values_init。
  • 排序注意事项：quick_kv_sort 对 MR_key_values 做原地交换（struct swap）。由于 MR_key_values 包含 pthread_mutex_t，在 Linux NPTL 实现下 pthread_mutex_t 是清零内存，realloc 的逐字节拷贝和排序中的 struct swap 在实际运行中安全。从标准严格角度看，包含 mutex 的结构体不应逐字节拷贝——可考虑将 mutex 改为指针或单独分配来规避，但在本实验中保持简洁。
