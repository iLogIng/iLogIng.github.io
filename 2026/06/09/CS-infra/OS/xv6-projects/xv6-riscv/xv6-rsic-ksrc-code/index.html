RISC-V 架构的三种模式（保留了一种）

Level Encode Name Abbreviation
0 00 User/Application 用户模式 U
1 01 Supervisor 监管者模式 S
2 10 Reserved 保留 -
3 11 Machine 机器模式 M

kernel.ld 链接器脚本

是一个 链接器脚本
描述链接器如何将编译产生的 多个目标文件(.o) 组合成一个可执行文件
各段依次排列
决定内核镜像的内存布局，约束生成的程序

ALIGN(exp) 内置函数，返回当前 位置计数器( . ) ，向上对齐到到 exp 的整数倍后的地址（通常是2的幂次）
保证后续的某个位置也满足对齐要求
位置计数器 会自动累加，并不是说 . = 0x80000000 就定义了 当前 . 的值，而是指 . 从 0x80000000 这个初始值累加，记录地址段大小
使用 ALIGN(exp) 就是返回将当前位置的 位置计数器 向上对齐后的值，通过 . = ALIGN(0x100) 将 . 更新为对齐后的计数器位置值

_trampoline 代码独占一内存页，单独对齐，在xv6中，trampline页会同时被内核页表和用户页表以 相同虚拟地址 映射，当 CPU 在这页上执行时，就可以安全地切换页表。如果不独占一页，就可能混入其他内核代码，导致切换时出现安全或正确性问题。
ASSERT 强制大小为一整页，防止生成有缺陷的内核

asm() 用于在C代码中嵌入汇编代码，即 内联汇编 技术。GCC编译器中 __asm()__ 与其等价。
volatile 关键字可以避免编译器优化内联的汇编代码
需要注意的是，内联的汇编代码通常与平台绑定，所以几乎无可移植性。

主程序 main.c

内核启动文件，是整个内核的入口函数

其内部结构很简单，只包含一对简单的 if-else 条件分支，并进行内核程序的直接初始化以及调度程序 scheduler() 的启用

其设计也很有意思，由于 xv6 是一个多核系统，则势必需要进行不同cpu核心的任务调度分配，
在 if 条件分支中，为了区分不同核心的不同动态，使用 cpuid() 调用，在主核心进行内核程序的初始化，
在 main.c 这一文件中还包含一个 volatile 修饰的静态全局变量 started ，使用 volatile 的作用是为了 **防止编译器优化 else 分支中的 while(started == 0) 这一忙等待循环。防止cpu读取寄存器中的缓存值，导致死循环。

cpuid() 在 kernel/defs.c 中声明，在 kernel/proc.c 中定义，

scheduler() 在 kernel/defs.c 中声明，在 kernel/proc.c 中定义
是 xv6 的核心调度循环，在启动后持续运行，不断在各个进程（或空闲状态）之间切换。每个hart都会进入该调度器。实际上，main.c程序在关机前都不会结束，这个就是整个 xv6 系统的主函数。

还可以看到 __atomic_thread_fence(__ATOMIC_SEQ_CST) 在设置 started 前后，保证在其前的 初始化操作都完成后 再将 started 赋值为1 (hart0 发布屏障)；在 忙等待 之后再进行 其它hart 的初始化。

保证了多核之间的内存可见性顺序 ，构成一个 无锁同步 模式

  • volatile 保障轮询的有效性(从内存中读取，以防止编译器优化，导致的寄存器缓存变量在忙等待中的死循环问题)
  • __atomic_thread_fence(__ATOMIC_SEQ_CST) 屏障保障数据的发布与订阅顺序（保证程序的逻辑顺序在编译时不受指令重排影响）。

volatile 关键字： 禁止编译器优化对变量的访问 ，表示不保证被修饰变量的值是否会在不知情的情况下被改变，在每次使用受 volatile 修饰的变量时都需要 从内存中 重新读取，而不使用寄存器中的缓存值。
注意： volatile 不能保证原子性 ，也 不保证顺序 ，仅能保证在使用时读取内存中的值

__atomic_thread_fence() 是 gcc 和 clang 提供的一个内置函数，用于生成一条硬件内存屏障指令，是 编译器内建的原子操作原语 ，
是编译器内置的 内存屏障 。强制编译器不会将屏障前后的访存指令跨屏障重排，同时会生成一条 CPU 硬件屏障指令，禁止硬件运行时对内存操作进行乱序执行，即保证在编译时指令重排操作不影响程序的逻辑顺序

hart 即 Hardware Thread(Tar T) 是 RSIC-V 指令集架构中的标准术语，表示 一个独立的硬件执行单元，或一个硬件线程
一个 CPU 核心 通常包含一套完整的运算单元
一个 hart 代表该核心上的一条独立指令执行流
大多数简单系统中，一个核心就是一个执行流，此时 一个核心 = 一个hart
若该线程支持超线程，一个物理核心可以同时运行多条指令流，此时 一个核心 = 多个hart

start.c

主要依赖 riscv.h 中riscv架构处理器寄存器操作函数、宏

  • start()
    • 启动程序，调起 main.c/main()
  • timerinit()
    • 初始化处理器时钟中断（定时器中断）

设置机器状态，启动分页，调起时钟中断初始化，返回到 main()

使用 w_mepc((uint64)main); 将 main() 函数的地址写入 mepc 寄存器，标识异常处理程序的起始地址

mepc ：是“机器异常程序计数器”（Machine Exception Program Counter）。
当 CPU 发生异常或中断而被迫跳转到 M-mode 的陷阱处理程序时，硬件会自动把被打断的指令地址（或下一条指令的地址）保存到 mepc 中。等内核处理完异常，执行 mret 指令时，CPU 就把 mepc 的值加载到普通 PC，从而实现精确返回。

mret : 机器模式返回寄存器 (Machine Return)，从机器模式的陷阱处理中返回

使用 asm volatile("mret"); 利用RISC-V的陷阱返回机制， mret 指令查看 mstatus 的 MPP (M Previous Privilege mode)字段，该字段已经通过以下方式设置

mret 检查 mepc 寄存器中的地址（即 main() 的地址）进行返回，跳转到 main.c/main() 函数

usertrap()

用户陷阱

在 trampoline.S/uservec 切换页表后调用，负责 分类处理所有来自用户态的陷阱 ，并最终返回用户态
只能被用户态触发

  • trampoline.S
    • uservec 陷阱向量调用
      • ld t0, 16(a0) … jalr t0 调用 p->trapframe->kernel_trap

处理来自于用户态的中断、异常或系统调用
从 trampoline.S 调用，并返回到 trampoline.S
返回值是用户寄存器(保存指向用户页表物理地址的指针) satp 供 trampoline.S 切换使用

satp 即 Supervisor Address Translation and Projection (监管者地址翻译与投影) 寄存器

  1. [核心] 使用 w_stvec((uint64)kernelvec) 更新 stvec 寄存器中的中断处理入口

    • 在我们身处内核态时，用于向 kerneltrap() 发送中断与异常，由 kernelvec 调起

  2. 在陷阱帧中保存用户的程序计数器

    • p->trapframe->epc = r_sepc(); 交由后续操作决定是否让出cpu

  3. [核心] 检查 scause (陷阱原因寄存器) 寄存器的值，根据 scause 分发处理

    • 系统调用
      • scause == 0
    • 设备中断
      • 记录 witch_dev
    • 缺页异常
      • scause == 15 加载页错误
      • scause == 13 存储页错误
      • 惰性分配
        • vmfault() 尝试处理惰性分配的页
        • 按需分配内存，避免提前分配，浪费内存
    • 预期外的陷阱

  4. 进程退出检查

    • if(killed(p)) kexit(-1);

  5. 设备定时器中断时让出cpu

    • if (witch_dev == 2) yield();
    • 触发定时中断，实现进程调度

  6. 准备返回用户态

    • prepare_return()

  7. 返回用户页表地址给 trampoline.S

    • 用户页可交换
    • uint64 satp = MAKE_SATP(p->pagetable); return satp;

  8. 注意，此时已经返回到 trampoline 中，直接按序向下执行 trampoline.S/userret 程序

使用汇编进行底层的上下文切换，C做高层的逻辑判断

kerneltrap()

内核陷阱

在 trampoline.S/kernelvec 切换页表后调用，负责 分类处理所有来自内核态的陷阱 ，并最终返回内核态
只能被内核态触发

  • kernelvec.S
    • kernelvec 陷阱向量程序调用
  • riscv.h
    • intr_get() 确定设备中断是否启用

在保存调用者保存的寄存器后，调起 kerneltrap()
kerneltrap() 返回内核中被中断的位置

  1. 获取对应寄存器的值

  2. 根据陷阱原因寄存器 scause 的值分发陷阱处理过程

    • 模式检测
      • (sstatus & SSTATUS_SPP) == 0
      • 检查是否通过监管者模式调起
    • [核心] 检测重复中断
      • intr_get() != 0
      • 防止内核陷阱处理过程本身被嵌套中断打断，内核陷阱必须在元自行的上下文中处理，避免竟态条件
    • 设备中断
      • 记录 witch_dev
    • 时钟中断
      • which_dev == 2 && myproc() != 0

  3. 直接返回 kernelvec

4份底层汇编文件

entry.S

xv6 系统真正的汇编入口，在每个CPU上都需要一个栈（在 start.c 中使用 __attribute__((aligned(16))) char stack0[4096 * NCPU]; 进行设置，定义了 stack0 ）
_entry 为每个hart独立运行

swtch.S

上下文交换程序
接口为：
void swtch(struct context *old, struct context *new);
保存当前的寄存器旧值，加载新的寄存器值

trampoline.S

依赖 risv.h
依赖 memlayout.h
依赖 trap.c

  • trap.c

    • usertrap(void)
    • prepare_return(void)

  • proc.h

    • 提供 struct trapframe 类型 p->trapframe变量
    • 提供 struct proc 类型 p变量

  • memlayout.h

    • #define TRAMPOLINE (MAXVA - PGSIZE)
    • #define TRAPFRAME (TRAMPOLINE - PGSIZE)

跳板汇编程序
用于处理 从用户空间到内核的陷阱的、从内核返回用户的 低级代码

内核将持有这段代码的页，
映射到用户和内核空间中的同一虚拟地址 (TRAMPOLINE)，
使得它在切换页表时能够继续工作。
kernel.ld 使此代码起始于一个页边界，并独立拥有该页

uservec 用户陷入内核态

userret 返回用户态

TRAPFRAME 在 任何进程的用户页表 中都指向进程自己的 trapframe 页面，而内核在创建进程时负责建立该映射。
trampoline 的代码只有一份，但被映射到所有进程的同一虚拟地址

kernelvec.S

内核态，不通过 call 调用，而是硬件在 产生陷阱时自动跳转 而来

  • trap.c
    • kerneltrap()

关于陷阱向量 uservec kernelvec ，都是由硬件进行调用，地址由 stvec 状态向量寄存器保存
在用户态时， stvec 为 uservec 地址
在内核态时， stvec 为 kernelvec 地址
通过 riscv.h/w_stvec() 在状态切换到边缘动态更新该寄存器，保证CPU在各模式下，陷阱都能进入正确的处理入口

从 xv6 的设计角度上，对于 内核态-用户态 模型， stvec 寄存器仅会保留 uservec kernelvec 这两种状态陷阱处理入口
当然，这两种陷阱入口的名字可由个人喜好进行设定，只要保证对相应状态的处理功能完备， 理论上也可以保存任意合法的陷阱入口地址
为什么仅有两种状态？因为上下文仅有 内核态、用户态 保证了设计的简洁性
